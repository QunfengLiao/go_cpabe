# 研究记录：CP-ABE 系统多租户基础能力

## 决策 1：当前租户上下文使用 `X-Tenant-Id`

**决策**：MVP 阶段使用请求头 `X-Tenant-Id` 表示当前租户，由后端在 `AuthRequired` 之后校验租户和成员关系。

**理由**：当前系统 Access Token 只表达登录身份，Refresh Token 只用于 Redis 刷新登录态。使用请求头可以最小化对双 Token 体系的侵入，用户切换租户时不需要重签 Token，也不需要把当前租户写入 Redis 会话。

**替代方案**：

- 将 `tenant_id` 写入 Access Token：切换租户需要重签 Token，并需要处理旧 Token 是否仍可访问旧租户的问题，MVP 成本较高。
- 将 `tenant_id` 写入 Redis Session：当前普通受保护请求不查 Redis，会把无状态 Access Token 改成半有状态模型。
- 每个接口请求体携带 `tenant_id`：容易被业务层误信任，不适合作为统一安全边界。

## 决策 2：新增独立 `roles` 表

**决策**：新增 `roles` 表，并用 `scope` 区分平台角色和租户角色。

**理由**：现有 `users.role` 是全局字符串，无法表达“同一用户在租户 A 是 DO，在租户 B 是 DU”。独立角色表可以让 `user_roles` 通过 `role_id` 关联，后续 RBAC 菜单权限也有扩展空间。

**替代方案**：

- 继续使用 `users.role`：无法表达多租户差异角色。
- 在 `user_roles` 中直接存 `role_code`：MVP 简单，但后续角色名称、描述、作用域和菜单权限扩展不清晰。

## 决策 3：保留 `users.role` 作为过渡兼容字段

**决策**：MVP 不删除 `users.role`，但明确它不再作为租户内授权依据。

**理由**：当前注册、登录、JWT Claims、前端资料展示和测试都依赖 `users.role`。直接删除会扩大改造范围，不符合“多租户基础能力”的阶段边界。

**替代方案**：

- 立即删除 `users.role`：会引发前后端和既有测试大面积改造。
- 同步维护 `users.role` 和 `user_roles` 并都作为授权依据：容易产生权限冲突，违反租户内授权边界。

## 决策 4：默认租户承接公开注册用户

**决策**：MVP 阶段公开注册用户自动加入 `default-tenant`；`data_owner` 映射为 `DO`，`data_user` 映射为 `DU`。

**理由**：当前系统已经支持公开注册，如果注册后不归属任何租户，用户登录后无法进入后续演示链路。默认租户能承接既有单租户体验，同时保留后续邀请制和租户选择能力。

**替代方案**：

- 注册后不加入任何租户：安全边界清晰，但当前演示体验会断开。
- 注册时强制选择或创建租户：需要新增公开租户创建或邀请码流程，超出 MVP。

## 决策 5：租户管理接口采用 `/api/v1/tenants`

**决策**：租户管理接口使用 `/api/v1/tenants`，当前用户租户接口使用 `/api/v1/me/tenants` 和 `/api/v1/me/switch-tenant`。

**理由**：现有后端 API 已统一挂在 `/api/v1`，用户资料接口是 `/api/v1/users/me`。`/me/tenants` 能表达“当前用户视角”，`/tenants` 能表达“管理资源视角”。

**替代方案**：

- 使用 `/api/v1/users/me/tenants`：与现有 `/users/me` 接近，但路径更长。
- 使用 `/api/me/switch-tenant`：与规格建议一致但缺少当前项目已有 `/api/v1` 前缀。

## 决策 6：数据库迁移同时保留 SQL 和 AutoMigrate

**决策**：新增 SQL 迁移脚本作为结构记录和部署依据，同时在现有启动入口中加入新增 Gorm 模型 AutoMigrate。

**理由**：项目当前已经同时存在 `backend/migrations/001_create_users.sql` 和 `db.AutoMigrate(&domain.User{})`。本计划沿用现状，避免突然引入第三方迁移工具。

**替代方案**：

- 只使用 AutoMigrate：缺少可审查的数据库结构文件。
- 引入迁移工具：更规范，但超出本功能范围，会改变项目运维方式。

## 决策 7：菜单字段 MVP 返回空数组

**决策**：租户切换和登录上下文可预留 `menus` 字段，但 MVP 返回空数组或省略，不实现菜单权限计算。

**理由**：用户已明确后续计划实现 Admin / DO / DU 菜单权限系统。本阶段只建立角色与租户上下文，不提前实现复杂 RBAC。

**替代方案**：

- 本阶段完整实现 RBAC 菜单：超出边界。
- 不预留菜单字段：后续前端切换租户后还要调整响应契约。
