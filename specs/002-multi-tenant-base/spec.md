# 功能规格：CP-ABE 系统多租户基础能力

**功能分支**：`feat/multi-tenant-base`
**创建日期**：2026-07-07
**状态**：草稿
**输入**：用户描述：“为当前 CP-ABE 文件加密与访问控制系统补充基于 `tenant_id` 的多租户基础能力，使不同组织、实验室或公司之间的数据隔离；同一用户可加入多个租户，并在不同租户下拥有不同角色。本阶段只做多租户基础，不实现完整 CP-ABE 加解密、复杂计费套餐资源配额，也不采用每租户一个数据库方案。”

## 用户场景与测试

### 用户故事 1：用户登录后获得租户列表（优先级：P1）

已注册用户登录系统后，可以看到自己所属的租户列表、每个租户下的角色，以及系统为单租户用户自动确定的当前租户。多租户用户可以在后续界面中选择当前工作租户，避免在错误组织上下文中操作数据。

**为什么是此优先级**：没有租户列表和当前租户，后续用户属性、文件、策略和审计日志都无法可靠归属到组织边界。

**独立测试**：创建一个用户并加入一个或多个租户，登录后检查响应中的租户列表、角色集合和当前租户结果。

**验收场景**：

1. **给定**用户只属于一个启用租户，**当**用户登录成功，**则**响应包含该租户信息，并自动设置 `currentTenantId`。
2. **给定**用户属于多个启用租户，**当**用户登录成功，**则**响应包含所有可用租户信息，并允许前端后续选择当前租户。
3. **给定**用户不属于任何启用租户，**当**用户登录成功，**则**系统返回可理解状态，提示用户暂无可用租户上下文，不能进入租户内业务页面。

---

### 用户故事 2：用户切换当前租户（优先级：P1）

已登录用户可以切换到自己所属且启用的租户。切换成功后，系统返回该租户下的角色和菜单权限预留信息，后续所有租户内数据查询都基于当前租户上下文过滤。

**为什么是此优先级**：同一用户在不同租户下可能拥有不同角色；必须先确定当前租户，才能正确判断该用户在当前组织中的可见数据和可用能力。

**独立测试**：让同一用户加入租户 A 和租户 B，并分配不同角色；调用切换租户入口后验证当前租户、角色和后续查询范围随之变化。

**验收场景**：

1. **给定**用户属于目标租户且租户启用，**当**用户切换到该租户，**则**系统切换成功并返回该租户下的角色集合和菜单权限预留信息。
2. **给定**用户不属于目标租户，**当**用户尝试切换，**则**系统拒绝切换，且不会暴露目标租户内数据。
3. **给定**目标租户已禁用，**当**用户尝试切换，**则**系统拒绝切换并提示租户不可用。

---

### 用户故事 3：管理员维护租户与成员关系（优先级：P2）

平台管理员或具备租户管理权限的管理员可以创建租户、查看租户、启用或禁用租户，并把用户加入或移出租户。MVP 阶段如果平台管理员能力尚未完整实现，则系统必须支持通过初始化数据建立默认租户和默认租户管理员。

**为什么是此优先级**：租户数据隔离需要可管理的租户与成员关系，否则系统无法演示多个组织之间的边界。

**独立测试**：创建两个租户，把同一用户分别加入两个租户并赋予不同角色，检查成员列表和用户登录响应是否正确。

**验收场景**：

1. **给定**管理员具备租户管理权限，**当**创建租户，**则**系统保存租户名称、编码、状态和描述，且租户编码唯一。
2. **给定**管理员把用户加入租户，**当**加入成功，**则**用户登录响应包含该租户。
3. **给定**管理员将用户移出租户，**当**用户再次请求该租户上下文，**则**系统拒绝该用户访问该租户内数据。

---

### 用户故事 4：同一用户在不同租户拥有不同角色（优先级：P2）

同一用户可以在租户 A 中是 DO，在租户 B 中是 DU。系统在判断菜单、接口调用和业务数据范围时，必须基于当前租户下的角色关系，而不是用户全局角色。

**为什么是此优先级**：原有单用户角色模型无法表达多组织协作场景，会导致跨租户权限污染。

**独立测试**：为同一用户在两个租户下分配不同角色，切换租户后检查角色、菜单预留信息和允许操作是否跟随当前租户变化。

**验收场景**：

1. **给定**用户在租户 A 是 DO、在租户 B 是 DU，**当**用户切换到租户 A，**则**系统按 DO 角色返回当前租户上下文。
2. **给定**同一用户切换到租户 B，**当**查询当前角色，**则**系统按 DU 角色返回当前租户上下文。
3. **给定**管理员给用户分配角色，**当**提交角色分配，**则**角色关系必须绑定到指定租户，不能只绑定到用户。

---

### 用户故事 5：租户内业务数据隔离（优先级：P1）

租户内用户查询属性、文件、访问策略、密钥记录、加解密记录和审计日志时，只能看到当前租户内的数据。用户即使知道其他租户的数据标识，也不能通过请求参数访问其他租户数据。

**为什么是此优先级**：多租户能力的核心价值是组织间数据隔离，这是后续文件加密共享主链路的安全基础。

**独立测试**：在两个租户中创建同名或相似业务数据，使用租户 A 用户尝试访问租户 B 数据，验证所有访问均被拒绝或返回空结果。

**验收场景**：

1. **给定**租户 A 用户处于租户 A 上下文，**当**查询用户属性、文件或策略，**则**只返回租户 A 数据。
2. **给定**租户 A 用户传入租户 B 的数据标识，**当**请求详情、更新或删除，**则**系统拒绝访问。
3. **给定**后续创建属性、文件、策略、密钥记录或审计日志，**当**保存数据，**则**数据必须归属于当前租户。

### 边界情况

- 用户属于多个租户，但其中部分租户被禁用或用户成员关系被禁用。
- 用户登录时没有任何启用租户，系统应允许识别账号身份，但不能进入租户内业务上下文。
- 请求缺少当前租户标识、当前租户标识格式非法或目标租户不存在。
- 用户尝试通过请求体、查询参数或请求头传入非成员租户标识访问数据。
- 用户在租户 A 拥有 Tenant Admin 角色，但在租户 B 仅为 DU，不能用租户 A 的权限管理租户 B。
- 同一用户被重复加入同一租户，系统应避免重复成员关系。
- 租户编码重复、租户名称为空、租户状态非法。
- 禁用租户后，该租户成员不能继续切换到该租户或访问租户内业务数据。
- 移除用户租户关系后，该用户已有当前租户上下文必须失效或在下一次请求时被拒绝。
- 初始化数据重复执行时，不应创建重复的默认租户、默认管理员和基础角色。

## 需求

### 功能需求

- **FR-001**：系统 MUST 支持租户实体，用于表示组织、实验室或公司边界；租户至少包含 `id`、`name`、`code`、`status`、`description`、`created_at`、`updated_at`，并 MAY 包含 `deleted_at`。
- **FR-002**：租户编码 `code` MUST 全局唯一；租户状态 MUST 至少支持启用和禁用。
- **FR-003**：系统 MUST 支持用户租户关系，用于表示同一用户加入了哪些租户；关系至少包含 `id`、`tenant_id`、`user_id`、`status`、`created_at`、`updated_at`。
- **FR-004**：同一用户 MUST 可以属于多个租户；同一用户在同一租户内 SHOULD 只有一条有效成员关系。
- **FR-005**：系统 MUST 将用户角色关系从“只绑定用户”调整为“绑定租户与用户”；角色分配至少包含 `id`、`tenant_id`、`user_id`、`role_id`、`created_at`、`updated_at`。
- **FR-006**：系统 MUST 支持基础角色 `TENANT_ADMIN`、`DO`、`DU`，并 SHOULD 预留 `PLATFORM_ADMIN` 角色。
- **FR-007**：原有 Admin 语义在租户内业务中 MUST 调整为 Tenant Admin；Platform Admin 是系统级角色，MVP 阶段可以只预留角色和数据结构。
- **FR-008**：登录成功后，系统 MUST 返回用户所属的启用租户列表；每个租户项至少包含租户标识、租户名称、租户编码和该用户在该租户下的角色列表。
- **FR-009**：当用户只属于一个启用租户时，系统 SHOULD 自动设置 `currentTenantId`；当用户属于多个启用租户时，系统 SHOULD 返回可选择租户列表，并允许前端要求用户选择当前租户。
- **FR-010**：系统 MUST 建立当前租户上下文能力；租户内受保护请求必须能同时确定 `user_id` 和 `currentTenantId`。
- **FR-011**：MVP 阶段当前租户上下文 SHOULD 通过请求头 `X-Tenant-Id` 传入，并由后端校验当前用户是否属于该租户；后续 MAY 在切换租户后把当前租户写入服务端会话或新访问凭证。
- **FR-012**：系统 MUST 在每个租户内业务请求中校验当前用户是否属于当前租户，且租户与成员关系均处于启用状态。
- **FR-013**：系统 MUST 提供切换租户能力；用户提交目标租户后，系统校验成员关系和租户状态，并返回该租户下的角色集合与菜单权限预留信息。
- **FR-014**：切换租户失败时，系统 MUST 保持原有登录身份有效，但不能建立目标租户上下文。
- **FR-015**：系统 MUST 提供租户管理基础能力，包括创建租户、查询租户列表、查询租户详情、启用租户、禁用租户、将用户加入租户、将用户移出租户、查询租户用户列表。
- **FR-016**：租户管理能力原则上 MUST 只允许 Platform Admin 或具备租户管理权限的管理员访问；如 MVP 阶段尚未实现 Platform Admin，系统 MUST 通过初始化数据提供默认租户和默认租户管理员。
- **FR-017**：给用户分配角色时，系统 MUST 限制在当前租户范围内操作，并 MUST 禁止把租户 A 的角色分配影响到租户 B。
- **FR-018**：系统 MUST 明确区分三类权限边界：多租户控制组织归属与数据隔离，RBAC 控制当前租户内菜单和接口，CP-ABE 控制属性私钥是否满足访问策略并能否解密密文文件。
- **FR-019**：系统 MUST NOT 使用租户成员关系或 RBAC 角色替代 CP-ABE 属性满足判断；DU 角色只能表示业务身份，不能自动代表可解密任意文件。
- **FR-020**：现有或后续创建的租户内业务数据实体 MUST 包含 `tenant_id` 并基于当前租户过滤；至少覆盖 `attributes`、`user_attributes`、`files`、`file_policies`、`encrypt_records`、`decrypt_records`、`audit_logs`。
- **FR-021**：如果上述业务表当前不存在，后续创建这些表时 MUST 从一开始包含 `tenant_id`，并在规格、计划和任务中继续保持该约束。
- **FR-022**：所有租户内业务查询、详情、更新、删除和统计 MUST 带当前租户过滤条件，禁止仅按业务数据标识查询。
- **FR-023**：系统 MUST 防止用户通过传入其他租户的 `tenant_id`、业务数据标识或组合参数访问非当前租户数据。
- **FR-024**：初始化数据 MUST 包含默认租户，例如 `default-tenant`；默认租户管理员；基础角色 `TENANT_ADMIN`、`DO`、`DU`；并可选预留 `PLATFORM_ADMIN`。
- **FR-025**：初始化数据 MUST 可重复执行且保持幂等，不应创建重复租户、重复角色或重复默认管理员关系。
- **FR-026**：系统 MUST 为租户、租户用户关系、租户角色关系和租户上下文校验提供可验证错误提示，区分未登录、无租户上下文、非租户成员、租户禁用和权限不足。
- **FR-027**：用户资料、登录响应和当前用户信息 MUST 逐步去除单一全局角色作为租户内授权依据；如需保留旧字段兼容，也必须标明其不能用于租户内业务授权。
- **FR-028**：后续查询当前用户菜单、角色、属性、文件、策略和审计日志时，系统 MUST 基于当前租户上下文返回结果。
- **FR-029**：本功能 MUST NOT 引入完整 CP-ABE 加解密、租户计费、套餐、资源配额、用户撤销、多授权机构、策略隐藏或每租户独立数据库。
- **FR-030**：所有新增规格、派生文档、业务说明和必要代码注释 MUST 使用简体中文；代码标识符、路径、命令、API 路径、JSON 字段、数据库表名、Go/TypeScript 类型名、第三方库和算法名称除外。

### 接口说明

- **用户登录**：`POST /api/v1/auth/login`。成功响应在现有登录态基础上增加 `currentTenantId` 和 `tenants`。示例字段为 `userId`、`email`、`currentTenantId`、`tenants[].tenantId`、`tenants[].tenantName`、`tenants[].tenantCode`、`tenants[].roles`。
- **切换租户**：`POST /api/v1/me/switch-tenant`。请求字段为 `tenantId`；成功后返回当前租户、该租户下角色集合和菜单权限预留信息。
- **创建租户**：管理员创建租户，提交租户名称、编码、状态和描述。
- **查询租户列表**：管理员查看可管理租户列表，普通租户用户只能看到自己所属租户。
- **查询租户详情**：管理员查看租户基础信息和状态。
- **启用或禁用租户**：管理员改变租户可用状态；禁用后该租户不可作为当前租户上下文。
- **加入租户**：管理员将用户加入指定租户，并可同时分配该租户下角色。
- **移出租户**：管理员将用户从指定租户移除或禁用成员关系。
- **查询租户用户列表**：管理员查看指定租户下成员、成员状态和租户内角色。

### 业务规则

- 多租户隔离是组织边界，不是菜单权限，也不是 CP-ABE 解密授权。
- RBAC 角色只在当前租户内生效；同一用户跨租户角色可以不同。
- CP-ABE 解密仍必须依赖后续真实属性私钥和访问策略满足判断，不能被租户角色绕过。
- 租户内数据创建时必须归属当前租户；租户内数据读取和变更时必须验证当前租户。
- Platform Admin 属于系统级角色，原则上可管理租户；Tenant Admin 只能管理当前租户范围内的用户、角色、属性、文件元数据和审计日志。
- DO 是当前租户内的数据拥有者，可上传文件、创建访问策略、加密文件、管理自己上传的文件。
- DU 是当前租户内的数据使用者，可查看可访问文件、下载密文并尝试解密。
- 默认租户用于承接既有单租户数据和演示环境，不代表生产环境组织模型。

### 宪章对齐

- **混合加密**：本功能只建立租户与数据隔离基础，不实现文件内容加密或数据密钥 DEK 封装；后续文件加密仍必须遵守“文件内容使用 AES-GCM，DEK 使用 RSA 或 CP-ABE 封装”。
- **真实 CP-ABE**：本功能不实现 CP-ABE 加解密，不得使用租户角色、租户属性或模拟逻辑冒充真实 CP-ABE 结果。
- **可插拔算法**：本功能不改变 `CryptoEngine` 边界；后续 Crypto、File、Policy、Benchmark、Audit 模块新增租户字段时，仍必须通过各自模块边界处理。
- **RSA 基线**：本功能不产生 RSA 与 CP-ABE 性能对比结论，不记录算法耗时。
- **策略表达与可视化**：本功能不实现访问树或 LSSS，但要求后续策略和属性数据必须带租户归属，避免跨租户解释策略。
- **可解释性**：系统必须能解释租户切换失败、租户禁用、非成员访问、跨租户数据拒绝、租户内角色不同导致菜单不同等结果。
- **模块边界**：租户管理和租户上下文能力可以作为独立 Tenant 能力或 User 模块扩展，但密码算法逻辑不得进入租户、用户、控制器或处理器。
- **语言规范**：本规格及后续 SpecKit 文档 MUST 使用简体中文。

### 关键实体

- **租户 Tenant**：组织、实验室或公司的数据隔离边界。关键属性包括 `id`、`name`、`code`、`status`、`description`、`created_at`、`updated_at`、`deleted_at`。
- **租户用户 TenantUser**：用户与租户的成员关系。关键属性包括 `id`、`tenant_id`、`user_id`、`status`、`created_at`、`updated_at`。
- **角色 Role**：租户内或系统级角色定义。MVP 包含 `TENANT_ADMIN`、`DO`、`DU`，预留 `PLATFORM_ADMIN`。
- **用户角色 UserRole**：用户在某个租户下的角色分配关系。关键属性包括 `id`、`tenant_id`、`user_id`、`role_id`、`created_at`、`updated_at`。
- **当前租户上下文 CurrentTenantContext**：一次受保护请求中被后端校验通过的当前租户标识、用户标识、租户内角色和成员状态。
- **租户内属性 Attribute**：后续 CP-ABE 属性管理数据，必须归属于租户。
- **租户内用户属性 UserAttribute**：后续用户属性分配数据，必须同时关联租户和用户。
- **租户内文件 File**：后续文件元数据和加密文件记录，必须归属于租户。
- **租户内访问策略 FilePolicy**：后续文件访问策略，必须归属于租户。
- **加密记录 EncryptRecord**：后续 DEK 封装、文件加密和性能记录，必须归属于租户。
- **解密记录 DecryptRecord**：后续解密尝试、成功或失败记录，必须归属于租户。
- **审计日志 AuditLog**：后续访问拒绝、解密成功、解密失败和管理操作日志，必须归属于租户。

### 非目标范围

- 不实现完整 CP-ABE 加密、解密、密钥生成或属性私钥分发。
- 不实现复杂租户计费、套餐、资源配额或用量限制。
- 不采用每个租户一个数据库、每个租户一套服务或每个租户一套独立表。
- 不实现完整 RBAC 菜单权限系统，只预留菜单权限返回结构和角色边界。
- 不实现文件上传、文件加密、访问策略解析、访问树可视化或 LSSS 可视化。
- 不实现用户撤销、多授权机构、策略隐藏、区块链审计等高级能力。
- 不承诺生产环境安全能力；如果用于生产环境，仍需专业安全审计和密钥管理审计。

## 成功标准

### 可衡量结果

- **SC-001**：测试人员可以在 5 分钟内创建两个租户，并把同一用户加入两个租户。
- **SC-002**：同一用户在两个租户下可被分配不同角色，切换租户后当前角色结果 100% 与该租户角色分配一致。
- **SC-003**：用户登录响应在 100% 的启用成员场景中返回正确租户列表；单启用租户用户自动获得 `currentTenantId`。
- **SC-004**：用户切换到所属且启用租户的成功率达到 100%；切换到非所属或禁用租户的拒绝率达到 100%。
- **SC-005**：跨租户访问属性、文件、策略、记录或审计日志的测试用例拒绝率达到 100%。
- **SC-006**：所有租户内业务查询和变更测试均能证明结果限定在当前租户范围内。
- **SC-007**：初始化流程重复执行 3 次后，默认租户、默认管理员关系和基础角色仍保持唯一且可用。
- **SC-008**：测试人员可以清楚区分多租户、RBAC 和 CP-ABE 三类权限边界，验收文档中不出现用角色替代 CP-ABE 解密权限的描述。

## 假设

- 既有用户注册和登录能力可复用，新增多租户能力在现有认证基础上扩展登录响应和受保护请求上下文。
- MVP 阶段使用共享数据库、共享表、`tenant_id` 隔离模型。
- MVP 阶段当前租户标识优先通过 `X-Tenant-Id` 请求头传入并由后端校验，计划阶段可评估是否在切换租户后签发包含当前租户的新访问凭证。
- 既有单用户全局角色字段可在过渡期保留，但不能继续作为租户内业务授权的唯一依据。
- 默认租户用于迁移和演示既有单租户数据，后续真实组织数据应显式归属于具体租户。
- Platform Admin 能力在 MVP 阶段只做角色和数据结构预留，完整平台后台可在后续功能中规划。
