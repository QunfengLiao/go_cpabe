# 研究记录：后端租户级 RBAC

## 决策 1：角色表演进而非新建 tenant_roles

**决策**：复用并演进现有 `roles` 表，增加 `tenant_id`、`scope_type`、`role_category`、`is_builtin`、`status` 等字段；不新建 `tenant_roles`。

**理由**：仓库扫描显示没有真实 `tenant_roles` 表，现有 `roles + user_roles` 已承担角色定义和授权职责。新建重复表会造成授权事实源分裂。

**替代方案**：新增 `tenant_roles`。被拒绝，因为会与现有 `roles`、`user_roles`、DTO `tenantRoles` 命名冲突，迁移复杂且容易产生双写。

## 决策 2：平台授权 tenant_id 从 NULL 收敛为 0

**决策**：迁移后 `user_roles.tenant_id=0` 表示平台授权，`tenant_id>0` 表示租户授权。

**理由**：当前 MySQL 唯一索引不会把 `NULL` 当作相等值，导致平台授权需要先查后写。改为 0 后，`UNIQUE(tenant_id, user_id, role_id)` 能完整约束平台和租户授权。

**替代方案**：继续使用 `NULL`。被拒绝，因为重复平台授权仍需特殊分支，迁移到状态化授权后更容易出现多条 active 记录。

## 决策 3：权限事实源只放数据库，不使用 Redis 缓存

**决策**：本阶段 AuthorizationService 每次按数据库事实源查询权限，不新增 Redis 权限缓存。

**理由**：角色撤销、过期、禁用和权限替换要求即时生效。当前项目没有成熟权限缓存失效机制，过早缓存会提高安全风险。

**替代方案**：缓存用户权限集合。被拒绝，因为需要额外失效协议，超出本阶段范围。

## 决策 4：最后管理员保护使用租户行锁

**决策**：成员角色替换、撤销管理员、移除成员、禁用影响管理员角色的操作，在事务内锁定当前 `tenants` 行，再统计和变更有效 `TENANT_ADMIN`。

**理由**：只锁单个 user_roles 行无法防止两个管理员并发互相移除。锁租户行能把同一租户治理角色变更串行化。

**替代方案**：只在更新后统计。被拒绝，因为并发事务可能同时看到变更前状态，最终产生 0 管理员。

## 决策 5：role code 创建后不可修改

**决策**：自定义角色创建后不允许修改 `code`，只允许改 `name`、`description` 和状态。

**理由**：role code 是权限排障、审计、导入导出和前端缓存的稳定标识。允许修改会破坏历史解释能力。

**替代方案**：允许未绑定成员的自定义角色修改 code。被拒绝，因为还需检查权限绑定、缓存和历史日志，复杂度不值得。

## 决策 6：旧角色判断保留为语义判断和兼容包装

**决策**：`HasRole`、`TENANT_ADMIN` 等角色判断不再用于新增功能授权，但保留用于最后管理员、内置角色分类、能力角色识别和旧接口兼容。

**理由**：RBAC 权限判断和角色语义是两类问题。完全删除角色判断会让“最后管理员”和“系统能力角色”这类规则难以表达。

**替代方案**：所有判断都改成 permission code。被拒绝，因为会丢失角色本身的治理语义。

## 决策 7：显式 SQL 迁移进入标准 migrate 入口

**决策**：新增幂等 SQL 迁移，并改造 `cmd/migrate` 执行显式迁移；`AutoMigrate` 不能单独承担本阶段变更。

**理由**：本阶段涉及唯一约束修改、`NULL` 回填、状态回填和数据种子，Gorm `AutoMigrate` 不能可靠表达这些变更顺序。

**替代方案**：只更新 Gorm 模型后执行 `AutoMigrate`。被拒绝，因为无法安全处理约束重建和数据回填。
