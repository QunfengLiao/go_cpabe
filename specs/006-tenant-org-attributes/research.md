# 研究记录：租户级组织架构与 CP-ABE 属性体系

## 决策 1：租户级属性字典独立于平台级 `policy_attributes`

**决策**：新增 `tenant_attributes` 和 `tenant_attribute_values`，访问策略构建器优先读取租户级属性字典。保留现有 `policy_attributes` 作为平台级策略管理兼容表，不在本功能中删除。

**理由**：当前 `policy_attributes` 是平台级静态属性来源，无法表达不同租户拥有不同部门树、枚举值和安全等级配置。租户级属性字典能让深信服科技、四川师范大学、香港友邦保险各自拥有不同 `department` 和可选值，同时不破坏已有访问策略管理页面。

**替代方案**：

- 直接给 `policy_attributes` 增加 `tenant_id`：会改变既有平台级语义，容易影响已经实现的 `PLATFORM_ADMIN` 管理能力。
- 继续使用静态前端 mock：不能满足真实租户组织与属性体系目标。

## 决策 2：部门角色采用“通用角色 + 部门作用域”

**决策**：部门角色记录在 `tenant_org_member_roles`，使用 `role_code` 表示 `ORG_MANAGER`、`ORG_MEMBER`、`DATA_OWNER`、`DATA_VISITOR`，并通过 `tenant_id + org_unit_id + user_id` 表达作用域。

**理由**：同一个通用角色可以在不同部门复用，避免 `AI_BG_MANAGER`、`CLOUD_BG_MANAGER` 等角色爆炸。该设计也让用户属性投影更简单：`department = AI_BG` 与 `org_role = ORG_MANAGER` 组合即可表达“AI BG 部门主管”。

**替代方案**：

- 为每个部门创建独立角色：角色数量随部门增长，难维护且不利于策略解释。
- 只用租户级 DO/DU 角色：无法表达部门内职责差异。

## 决策 3：组织树使用邻接表 + 稳定路径

**决策**：`tenant_org_units` 同时保存 `parent_id`、`path`、`level` 和 `sort_order`。`parent_id` 负责树结构，`path` 负责快速解释和策略展示。

**理由**：邻接表容易用 Gorm 维护，适合当前演示规模；稳定路径让访问策略构建器可以展示“属于某部门及其下级”的语义。保存策略时以 `valueId/org_unit_id` 为权威，`path` 作为解释字段，降低组织移动造成的历史策略歧义。

**替代方案**：

- 只保存 `parent_id`：每次构造路径都要递归，解释能力弱。
- 闭包表：适合大规模树查询，但本阶段复杂度过高。

## 决策 4：用户 CP-ABE 属性以同步表保存

**决策**：新增 `user_attributes` 保存用户在租户内的有效 CP-ABE 属性，字段包含 `attr_code`、`value_id/value_code/value_path`、`number_value`、`source_type/source_id` 和状态。

**理由**：访问策略匹配和后续密钥发放需要稳定、可解释的属性集合。同步表能记录属性来源，便于解释“属性来自部门成员、部门角色、租户角色还是演示安全等级”，也避免每次匹配都临时聚合多张表。

**替代方案**：

- 每次解密前实时 JOIN 组织和角色：逻辑复杂，难解释历史结果。
- 只把属性保存在前端：不可信且无法服务后续后端策略匹配。

## 决策 5：构建器接口返回复合属性元数据

**决策**：`GET /tenants/:id/access-policy/attributes` 返回属性定义、允许操作符、枚举值和部门树。`department` 是 `tree` 类型，`org_role/tenant_role/data_category` 是 `enum` 类型，`security_level` 是 `number` 类型。

**理由**：前端构建器需要一次加载当前租户的完整属性上下文，减少多次请求和跨租户缓存风险。复合返回也能直接驱动不同控件：树形选择器、枚举选择器、数字比较输入。

**替代方案**：

- 前端分别请求属性、部门树和枚举值：可行但状态同步更复杂，MVP 优先一次返回。
- 后端只返回平铺字符串数组：无法保存 `valueId/path`，不满足稳定引用要求。

## 决策 6：MVP 不新增 Redis 缓存

**决策**：组织树和属性字典先直接查 MySQL；Redis 只沿用现有登录态能力。

**理由**：演示数据规模小，直接查询足以满足用户感知性能。新增缓存会引入失效复杂度，尤其是组织移动、属性禁用和租户切换场景。

**替代方案**：

- 缓存组织树和属性字典：后续可做，但需要设计失效策略。

## 决策 7：Go 注释作为计划和任务硬要求

**决策**：后续实现所有新增 Go 函数/方法、导出类型、实体字段、Handler、Service、Repository 均按项目宪章补充中文注释；tasks 必须加入“关键注释和可读性检查”。

**理由**：本功能涉及租户隔离和访问控制输入，错误实现会影响策略和解密判断。中文注释需要解释业务边界和安全边界，而不是复述代码。

**替代方案**：

- 只给导出函数写注释：不满足宪章。
