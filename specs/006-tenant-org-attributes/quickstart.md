# 快速验证指南：租户级组织架构与 CP-ABE 属性体系

## 前置条件

- MySQL 和 Redis 按当前项目方式启动。
- 已存在三个演示租户：四川师范大学、深信服科技、香港友邦保险。
- 已完成本功能迁移和种子数据初始化。
- 至少准备以下账号：
  - 一个租户管理员。
  - 一个 DATA_OWNER。
  - 一个 DATA_VISITOR。

## 迁移与启动

```powershell
cd D:\04_code\go_cpabe\backend
```


## 验证 1：组织树按租户隔离

1. 以深信服科技租户用户登录并切换到深信服科技。
2. 调用或打开组织树入口，确认包含安全 BG、云 BG、AI BG 等节点。
3. 切换到四川师范大学，确认组织树变为计算机科学学院、教务处、图书馆等节点。
4. 尝试用当前租户上下文读取另一个租户的组织树，应被拒绝。

**期望结果**：不同租户组织树完全不同，跨租户读取失败。

## 验证 2：构建器加载真实属性字典

1. 以 DATA_OWNER 进入访问策略构建器。
2. 确认左侧属性字典来自接口，不再显示静态“研发部/财务部/安全部”假数据。
3. 检查 `department` 使用树形选择器。
4. 检查 `org_role`、`tenant_role`、`data_category` 使用枚举选择。
5. 检查 `security_level` 支持 `>=`、`<=`、`=`。

**期望结果**：属性字典随租户变化，控件类型随属性类型变化。

## 验证 3：保存稳定策略节点

1. 在深信服科技构建策略：`(department 属于 AI_BG AND org_role = ORG_MANAGER) OR tenant_role = TENANT_ADMIN`。
2. 保存并重新打开策略。
3. 查看 JSON 预览，确认部门节点包含稳定 `valueId/valueCode/label/path`。
4. 切换到四川师范大学后重新进入构建器，确认不显示深信服科技 AI BG。

**期望结果**：策略节点不只保存中文名，租户切换后没有旧租户数据残留。

## 验证 4：部门成员和部门角色

1. 以租户管理员将某用户加入 AI BG。
2. 为该用户设置 `ORG_MANAGER` 和 `DATA_OWNER`。
3. 重复保存同样角色。
4. 查询部门成员。

**期望结果**：角色绑定保持幂等，同一用户同一部门不产生重复 `ORG_MANAGER`。

## 验证 5：同步用户 CP-ABE 属性

1. 对上述用户执行用户属性同步。
2. 查看同步结果。
3. 确认包含 `department = AI_BG`、`org_role = ORG_MANAGER`、租户角色和安全等级属性。
4. 将用户从 AI BG 移除后再次同步。

**期望结果**：旧部门属性失效，不再作为有效用户属性返回。

## 验证 6：权限边界

1. 平台管理员尝试维护具体部门，应被拒绝或没有入口。
2. DATA_OWNER 尝试维护组织结构，应被拒绝。
3. DATA_VISITOR 尝试读取他人属性，应被拒绝。
4. 任意用户尝试跨租户读取组织、成员、属性或用户属性，应被拒绝。

**期望结果**：所有越权操作拒绝率为 100%。

## 非目标确认

验收本功能时不检查以下内容：

- 真实 CP-ABE 加解密。
- 完整文件加密上传。
- 企业微信或外部组织同步。
- 复杂审批流。
- 复杂审计后台。
