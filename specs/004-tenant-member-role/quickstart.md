# 快速验证指南：租户成员角色分配

## 前置条件

- 已完成多租户基础能力、平台管理员能力、租户成员列表能力。
- 数据库中存在至少一个启用租户。
- 租户中至少有一个 `TENANT_ADMIN` 和一个普通成员。
- `roles` 中存在 `TENANT_ADMIN`、`DO`、`DU`、`PLATFORM_ADMIN`。

## 后端验证

### 1. 租户管理员分配数据拥有者

1. 使用租户管理员登录，获取访问凭证。
2. 调用：

```http
PUT /api/v1/tenants/1/members/2/role
Content-Type: application/json

{
  "roleCode": "DATA_OWNER"
}
```

3. 期望返回成功，成员 `roles` 包含 `DO`。
4. 查询成员列表，确认该成员显示数据拥有者。

### 2. 租户管理员分配数据访问者

1. 使用同一租户管理员调用：

```http
PUT /api/v1/tenants/1/members/2/role
Content-Type: application/json

{
  "roleCode": "DATA_VISITOR"
}
```

2. 期望返回成功，成员 `roles` 包含 `DU`。
3. 数据库中该用户在该租户下不再保留 `DO`。

### 3. 平台管理员不能调用普通角色分配接口

1. 使用只有 `PLATFORM_ADMIN`、不是目标租户 `TENANT_ADMIN` 的用户登录。
2. 调用普通角色分配接口。
3. 期望返回 403 或明确业务错误，提示平台管理员不参与租户内业务角色分配。

### 4. 普通成员不能调用普通角色分配接口

1. 使用普通成员登录。
2. 调用普通角色分配接口。
3. 期望返回 403 或无权限错误。

### 5. 跨租户操作被拒绝

1. 使用租户 A 的 `TENANT_ADMIN` 登录。
2. 尝试给租户 B 的成员分配角色。
3. 期望返回无权限或目标成员不属于该租户。

### 6. 幂等和事务验证

1. 对同一成员重复保存 `DATA_OWNER` 多次。
2. 查询 `user_roles`，确认同一租户、同一用户、同一普通业务角色没有重复有效记录。
3. 使用测试桩或故障注入模拟写入失败，确认旧角色没有被永久删除。

## 前端验证

1. 使用租户管理员登录并进入租户成员列表。
2. 确认成员列表显示“角色”列。
3. 点击普通成员行的“分配角色”或“修改角色”按钮。
4. 确认弹窗显示昵称、邮箱、当前角色。
5. 确认弹窗只包含“数据拥有者”和“数据访问者”选项。
6. 保存成功后弹窗关闭，成员列表刷新并显示新角色。
7. 使用平台管理员查看成员列表，确认普通业务角色分配按钮不展示或为禁用态，并提示平台管理员不参与租户内业务角色分配。

## 通过标准

- 后端所有角色分配、越权、幂等、事务测试通过。
- 前端类型检查和构建通过。
- 成员列表能立即回显最新角色。
- 数据库不出现重复普通业务角色记录。
- 代码注释检查通过：新增或修改的 Go 函数/方法都有中文前置注释，权限和事务逻辑有块级解释。
