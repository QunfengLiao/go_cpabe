# 研究记录：Platform Admin 平台管理员租户管理能力

## 决策 1：平台后台接口使用 `/api/v1/platform`

**决策**：新增 `/api/v1/platform` 路由组承载平台后台能力，例如 `/api/v1/platform/tenants`、`/api/v1/platform/dashboard`。

**理由**：当前 `/api/v1/tenants` 同时承担平台管理和当前用户租户视角，容易把 Tenant Admin 的租户内管理能力与 Platform Admin 的平台治理能力混在一起。独立平台路由组能让权限中间件、前端菜单和接口语义都更清楚。

**替代方案**：

- 继续使用 `/api/v1/tenants`：改动少，但会延续平台/租户权限混杂问题。
- 使用 `/api/platform`：接近用户建议，但当前项目所有接口都在 `/api/v1` 下，破坏版本前缀一致性。

## 决策 2：平台授权只认 `PLATFORM_ADMIN`

**决策**：平台接口必须通过 `PlatformAdminRequired` 中间件，校验 `user_roles.tenant_id IS NULL` 且角色为 `PLATFORM_ADMIN`。

**理由**：当前 `ensurePlatformOrLegacyAdmin` 允许旧 `users.role = admin`，这会让旧单租户管理员获得平台能力，不符合规格要求。平台授权必须与租户上下文无关，并只由平台级角色决定。

**替代方案**：

- 继续允许旧 `admin`：兼容性好，但会导致 Tenant Admin 和历史 Admin 越权。
- 在 JWT 中写入平台角色：可减少查询，但角色变化后旧 token 仍可能继续有效，且当前系统尚未将细粒度角色写入 token。

## 决策 3：角色模型复用现有 `roles.scope` 和 `user_roles.tenant_id`

**决策**：不新增角色表字段。`roles.scope = platform/tenant` 区分作用域；平台角色 `tenant_id = NULL`，租户角色 `tenant_id` 必填。

**理由**：当前迁移和领域模型已经具备该结构，符合多租户基础规格，也符合本功能需求。复用能减少迁移风险，并让后续 RBAC 菜单扩展保留空间。

**替代方案**：

- 单独创建 `platform_user_roles`：可更强约束平台角色唯一性，但会增加表和仓储复杂度。
- 继续使用 `users.role`：无法表达同一用户在多个租户下不同角色，也无法表达平台级和租户级授权边界。

## 决策 4：平台级角色重复分配先做仓储层幂等保护

**决策**：本阶段先在 repository/service 中实现 `EnsurePlatformRole` 的先查后写幂等逻辑；数据库层唯一约束作为后续加强项。

**理由**：MySQL 唯一索引对 `NULL` 的处理无法阻止多个 `(NULL, user_id, role_id)`。立即增加生成列或拆表会扩大迁移范围。仓储层幂等足以满足演示和当前验收，但计划中必须注明风险和后续加强方案。

**替代方案**：

- 新增生成列 `tenant_scope_key = COALESCE(tenant_id, 0)` 并建唯一索引：数据库约束更强，但需要额外迁移和兼容性验证。
- 拆分平台角色表：约束清晰，但与已有 `user_roles` 设计重复。

## 决策 5：初始化命令负责创建或分配首个 Platform Admin

**决策**：扩展 `backend/cmd/admin/main.go`，新增受控命令创建平台管理员或给已有账号分配 `PLATFORM_ADMIN`。

**理由**：公开注册不能创建平台管理员。当前项目已有本地受控管理员命令，适合承担首个高权限账号初始化。该命令可以复用现有配置、数据库连接、密码哈希和幂等初始化流程。

**替代方案**：

- 通过前端页面创建首个 Platform Admin：需要先有平台管理员才能访问平台页面，形成启动悖论。
- 手写 SQL 插入：可行但不利于跨环境和幂等验证。

## 决策 6：登录响应增加平台角色信息用于前端菜单

**决策**：登录响应在现有 `tenants` 外增加 `platform_roles` 或等价字段；前端据此设置 `isPlatformAdmin` 并展示平台菜单。

**理由**：前端目前只有旧 `user.role` 和租户列表，无法可靠判断平台级角色。菜单隐藏是体验需求，安全仍由后端平台中间件保证。

**替代方案**：

- 前端调用单独 `/platform/me` 判断：角色变化实时性更好，但登录后多一次请求。
- 使用旧 `user.role = admin` 判断：不符合平台级角色模型。

## 决策 7：平台控制台轻量实现，平台审计预留

**决策**：平台控制台统计建议本阶段实现轻量接口；平台审计日志本阶段只预留设计和调用点，不强制落表。

**理由**：统计项可直接由现有表计算，实现成本低，能提升平台后台完整度。审计日志当前没有基础表和模块，强行落地会扩大范围，且宪章要求 Audit 模块应有清晰边界。

**替代方案**：

- 控制台和审计都不实现：主链路更快，但前端平台首页会空。
- 审计立即建表：可演示操作日志，但会引入新的 Audit 模块设计和迁移范围。

## 决策 8：前端使用单一布局和菜单配置，不复制页面

**决策**：在现有 `AppLayout` 内增加角色驱动菜单配置；Platform Admin 看到平台后台入口，非 Platform Admin 不展示。

**理由**：项目已有单一登录页、多租户登录配置和统一布局。复制多套页面会提高维护成本，也违背规格中“不复制三套登录页”的要求。

**替代方案**：

- 为平台后台新建完全独立布局：视觉上可隔离，但当前应用规模下会重复账号切换、主题、退出等逻辑。
- 仅靠后端 403，不做前端菜单控制：安全可行，但用户体验差。
