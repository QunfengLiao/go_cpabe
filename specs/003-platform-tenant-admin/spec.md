# 功能规格：Platform Admin 平台管理员租户管理能力

**功能分支**：`feat/platform-tenant-admin`  
**创建日期**：2026-07-07  
**状态**：草稿  
**输入**：用户描述：“实现 Platform Admin 平台管理员，用于管理平台下的租户、租户用户关系和租户管理员分配；明确 Platform Admin、Tenant Admin、DO、DU 的权限层级和安全边界；本阶段只做平台级租户管理，不实现 CP-ABE 加解密、完整租户内 RBAC 菜单、文件上传下载、策略加密、解密记录等业务。”

## 用户场景与测试

### 用户故事 1：平台管理员管理租户生命周期（优先级：P1）

Platform Admin 登录平台后，可以查看所有租户，创建新租户，查看租户详情，并启用或禁用租户。平台管理员需要能够维护四川师范大学、深信服科技、香港友邦保险等演示租户，也能继续创建新的组织租户。

**为什么是此优先级**：租户是多租户系统的组织边界。没有平台级租户生命周期管理，就无法可靠维护不同组织的数据隔离入口。

**独立测试**：为用户分配平台管理员身份后，验证该用户可以完成租户列表查询、创建租户、查看租户详情、启用租户和禁用租户；同时验证非平台管理员无法访问这些平台管理能力。

**验收场景**：

1. **给定**用户拥有 `PLATFORM_ADMIN` 平台级角色，**当**用户查询租户列表，**则**系统返回平台下所有租户及其状态。
2. **给定**用户拥有 `PLATFORM_ADMIN` 平台级角色，**当**用户提交合法租户名称和唯一租户编码，**则**系统创建启用状态的租户。
3. **给定**租户编码已经存在，**当**平台管理员再次使用相同编码创建租户，**则**系统拒绝创建并返回明确的重复编码提示。
4. **给定**租户处于启用状态，**当**平台管理员禁用该租户，**则**普通租户内用户和 Tenant Admin 均不能再进入该租户工作台。
5. **给定**租户处于禁用状态，**当**平台管理员查看该租户详情，**则**系统仍允许查看租户基础信息和历史关系。

---

### 用户故事 2：平台管理员维护租户用户关系（优先级：P1）

Platform Admin 可以查看某个租户下的用户列表，将已有用户加入指定租户，也可以将用户移出指定租户。加入租户后，用户才能在该租户内拥有角色和后续业务上下文。

**为什么是此优先级**：租户用户关系是多租户授权的基础。平台管理员必须能够明确控制哪些用户属于哪个组织，避免用户因为历史登录态或旧租户上下文进入错误租户。

**独立测试**：准备一个已存在用户和一个已存在租户，使用 Platform Admin 将用户加入租户后，验证用户租户列表包含该租户；再移出用户后，验证用户不能切换进入该租户。

**验收场景**：

1. **给定**目标用户和目标租户都存在，**当**平台管理员将用户加入租户，**则**系统建立有效租户用户关系。
2. **给定**用户已经属于该租户，**当**平台管理员再次加入同一用户，**则**系统不创建重复关系，并返回可理解结果。
3. **给定**目标用户不存在，**当**平台管理员尝试加入租户，**则**系统拒绝操作并提示用户不存在或参数无效。
4. **给定**用户已经被移出租户，**当**该用户尝试切换进入该租户，**则**系统拒绝进入租户上下文。

---

### 用户故事 3：平台管理员分配租户管理员（优先级：P1）

Platform Admin 可以给某个租户下的用户分配 `TENANT_ADMIN` 角色，也可以移除用户在该租户下的 `TENANT_ADMIN` 角色。Tenant Admin 只能管理当前租户内部用户、角色、属性和审计等内容，不能管理平台租户，也不能管理其他租户。

**为什么是此优先级**：平台管理员负责组织边界，租户管理员负责租户内部治理。两者职责分离后，平台管理不会变成“万能解密”或跨租户业务权限。

**独立测试**：将用户加入租户后分配 `TENANT_ADMIN`，验证该用户可以管理当前租户内用户；再验证该用户不能访问平台租户管理能力，也不能管理其他租户。

**验收场景**：

1. **给定**用户已经属于目标租户，**当**平台管理员分配 `TENANT_ADMIN`，**则**系统为该用户建立当前租户内的租户管理员角色。
2. **给定**用户不属于目标租户，**当**平台管理员尝试分配 `TENANT_ADMIN`，**则**系统应拒绝并提示需先加入租户。
3. **给定**用户已经是目标租户的 `TENANT_ADMIN`，**当**平台管理员再次分配同一角色，**则**系统不创建重复角色关系。
4. **给定**用户仅拥有某租户的 `TENANT_ADMIN`，**当**用户访问平台租户管理能力，**则**系统拒绝访问。
5. **给定**用户在租户 A 是 `TENANT_ADMIN`，**当**用户尝试管理租户 B，**则**系统拒绝跨租户管理。

---

### 用户故事 4：平台管理接口只接受平台级授权（优先级：P1）

平台管理能力必须基于平台级角色判断权限，不依赖当前租户上下文。普通 Tenant Admin、DO、DU 即使已经登录并拥有某个租户上下文，也不能访问平台管理能力。

**为什么是此优先级**：平台接口如果依赖当前租户或旧全局 Admin 语义，容易导致租户管理员越权管理平台或其他租户。

**独立测试**：分别使用 Platform Admin、Tenant Admin、DO、DU 身份访问平台管理能力，验证只有 Platform Admin 成功，其余身份均被拒绝。

**验收场景**：

1. **给定**用户拥有平台级 `PLATFORM_ADMIN`，**当**访问平台管理能力，**则**系统允许访问。
2. **给定**用户仅拥有租户级 `TENANT_ADMIN`，**当**访问平台管理能力，**则**系统拒绝访问。
3. **给定**用户仅拥有租户级 `DO` 或 `DU`，**当**访问平台管理能力，**则**系统拒绝访问。
4. **给定**平台管理请求没有当前租户标识，**当**用户拥有 `PLATFORM_ADMIN`，**则**系统仍能完成平台级授权判断。

---

### 用户故事 5：禁用租户后的访问边界（优先级：P1）

租户被禁用后，普通租户内用户、Tenant Admin、DO、DU 都不能切换进入该租户或访问该租户工作台。Platform Admin 仍然可以在平台后台查看该租户信息和历史关系，但禁用操作不删除历史数据。

**为什么是此优先级**：禁用租户是平台级治理能力，必须立即阻断租户内访问，同时保留平台管理和审计可追溯性。

**独立测试**：禁用一个已有成员和 Tenant Admin 的租户，验证该租户成员无法进入工作台；再使用 Platform Admin 查询租户详情和用户关系，验证仍可查看。

**验收场景**：

1. **给定**租户已禁用，**当**该租户内 Tenant Admin 尝试进入租户工作台，**则**系统拒绝进入。
2. **给定**租户已禁用，**当**该租户内 DO 或 DU 尝试进入租户工作台，**则**系统拒绝进入。
3. **给定**租户已禁用，**当**Platform Admin 查看租户详情，**则**系统允许查看。
4. **给定**租户已禁用，**当**系统展示租户列表，**则**仍保留该租户并明确显示禁用状态。

---

### 用户故事 6：平台控制台和平台审计预留（优先级：P3）

Platform Admin 可以看到平台控制台入口，用于展示租户总数、启用租户数、禁用租户数、平台用户总数、租户用户关系数量、Tenant Admin 数量等统计。平台级操作日志可以先预留能力，后续再补全完整审计。

**为什么是此优先级**：控制台和审计有助于演示平台治理价值，但不会阻塞租户管理主链路。

**独立测试**：验证平台后台存在控制台入口或预留位置；如果统计能力已实现，则统计数字与当前数据一致；如果审计未实现，则规格和后续计划明确其为预留能力。

**验收场景**：

1. **给定**Platform Admin 登录平台后台，**当**进入平台首页，**则**系统展示平台控制台入口。
2. **给定**统计能力已实现，**当**平台管理员查看控制台，**则**租户、用户和租户管理员统计与实际数据一致。
3. **给定**平台审计尚未完整实现，**当**规划后续任务，**则**创建租户、启用租户、禁用租户、用户加入租户、用户移出租户、分配和移除 Tenant Admin 均被列为审计候选事件。

### 边界情况

- 租户编码为空、名称为空、编码包含大写字母、空格、下划线或其他不允许字符。
- 租户编码与已有租户重复，包括演示租户 `scnu`、`sangfor`、`aia-hk`。
- 平台管理员尝试禁用不存在的租户。
- 平台管理员将不存在的用户加入租户。
- 平台管理员将用户加入不存在或已禁用的租户。
- 平台管理员重复加入同一用户到同一租户。
- 平台管理员给不属于目标租户的用户分配 `TENANT_ADMIN`。
- 平台管理员重复分配同一用户在同一租户下的同一角色。
- 平台管理员移除某租户最后一个有效 `TENANT_ADMIN`。
- Tenant Admin 尝试创建、启用、禁用其他租户。
- DO 或 DU 尝试访问平台管理能力。
- Platform Admin 尝试查看主密钥明文、用户私钥明文、租户文件明文或绕过 CP-ABE 访问策略。
- 用户本地或会话中保留旧租户上下文时，平台级权限判断仍必须以服务端平台角色为准。

## 需求

### 功能需求

- **FR-001**：系统 MUST 支持 `PLATFORM_ADMIN` 平台级角色，用于平台租户治理能力授权。
- **FR-002**：系统 MUST 保持 `TENANT_ADMIN`、`DO`、`DU` 为租户级角色，用于当前租户内的管理、数据拥有和数据使用身份表达。
- **FR-003**：系统 MUST 区分角色作用域；`PLATFORM_ADMIN` 的作用域为平台级，`TENANT_ADMIN`、`DO`、`DU` 的作用域为租户级。
- **FR-004**：系统 MUST 支持平台级角色不绑定具体租户；租户级角色必须绑定具体租户。
- **FR-005**：系统 MUST 能给指定用户分配 `PLATFORM_ADMIN`，并保证该角色不会自动赋予任何租户文件解密能力。
- **FR-006**：只有拥有 `PLATFORM_ADMIN` 的用户 MAY 访问平台管理能力。
- **FR-007**：Tenant Admin、DO、DU MUST NOT 访问平台管理能力，即使其拥有有效登录态或当前租户上下文。
- **FR-008**：平台管理能力 MUST NOT 依赖当前租户上下文完成授权；授权依据必须是平台级角色。
- **FR-009**：系统 MUST 支持查询平台下全部租户列表，列表至少能表达租户名称、租户编码、租户状态和基础描述。
- **FR-010**：系统 MUST 支持创建租户，创建输入至少包含租户名称、租户编码和租户描述；未显式指定状态时，租户默认启用。
- **FR-011**：创建租户时，租户名称和租户编码 MUST 不能为空。
- **FR-012**：创建租户时，租户编码 MUST 全局唯一。
- **FR-013**：租户编码 SHOULD 只允许小写字母、数字和中划线，并应拒绝包含其他字符的编码。
- **FR-014**：创建租户遇到重复编码时，系统 MUST 返回明确错误，方便用户知道需要更换编码。
- **FR-015**：系统 MUST 支持查看租户详情，包括租户基础信息、状态和可用于平台治理的关系摘要。
- **FR-016**：系统 MUST 支持启用租户和禁用租户。
- **FR-017**：禁用租户后，Tenant Admin、DO、DU MUST NOT 切换进入该租户或进入该租户工作台。
- **FR-018**：禁用租户后，Platform Admin MUST 仍可在平台后台查看该租户信息。
- **FR-019**：禁用租户 MUST NOT 删除租户历史数据、成员关系、角色关系或后续审计记录。
- **FR-020**：系统 MUST 支持查询指定租户下的用户列表，列表至少能表达用户基础信息、成员状态和租户内角色。
- **FR-021**：系统 MUST 支持 Platform Admin 将已存在用户加入指定租户。
- **FR-022**：将用户加入租户时，系统 MUST 校验用户存在、租户存在，并避免同一用户在同一租户下产生重复有效成员关系。
- **FR-023**：系统 SHOULD 对已禁用租户的新增成员操作进行保护；默认规则为拒绝向禁用租户新增普通成员，除非后续计划明确需要特殊迁移流程。
- **FR-024**：系统 MUST 支持 Platform Admin 将用户移出指定租户。
- **FR-025**：用户移出租户后，系统 MUST 阻止该用户继续使用该租户作为当前工作租户。
- **FR-026**：移出租户用户时，系统 SHOULD 防止租户失去最后一个有效 `TENANT_ADMIN`；如需允许，必须提供明确确认或后续恢复路径。
- **FR-027**：系统 MUST 支持 Platform Admin 给指定租户下的用户分配 `TENANT_ADMIN`。
- **FR-028**：分配 `TENANT_ADMIN` 时，目标用户 MUST 已属于该租户；若不属于，应拒绝并提示先加入租户。
- **FR-029**：同一用户在同一租户下 MUST NOT 重复拥有同一个租户级角色。
- **FR-030**：系统 MUST 支持 Platform Admin 移除指定用户在指定租户下的 `TENANT_ADMIN`。
- **FR-031**：Tenant Admin MUST 只能管理当前租户内部用户、角色、属性、审计等租户内资源，不能创建、启用、禁用或管理其他租户。
- **FR-032**：DO MUST 仅表示当前租户内数据拥有者身份，可用于后续上传文件、创建访问策略、加密文件和管理自己上传文件的业务规划。
- **FR-033**：DU MUST 仅表示当前租户内数据使用者身份，可用于后续查看属性、下载密文和尝试解密文件的业务规划。
- **FR-034**：本功能 MUST NOT 实现文件上传、文件下载、策略加密、解密记录、完整 CP-ABE 加解密或完整租户内 RBAC 菜单权限。
- **FR-035**：Platform Admin MUST NOT 因平台角色而查看主密钥明文、用户私钥明文、密码明文或任何可绕过访问策略的敏感秘密。
- **FR-036**：Platform Admin MUST NOT 因平台角色而绕过 CP-ABE 访问策略下载或查看明文文件。
- **FR-037**：系统返回给平台管理界面的数据 MUST 避免包含主密钥、用户私钥、密码、明文文件内容等敏感信息。
- **FR-038**：平台控制台 MAY 展示租户总数、启用租户数、禁用租户数、平台用户总数、租户用户关系数量和 Tenant Admin 数量。
- **FR-039**：平台级审计 MAY 记录创建租户、启用租户、禁用租户、用户加入租户、用户移出租户、分配 Tenant Admin 和移除 Tenant Admin 等操作；如果本阶段不实现完整审计，必须在计划中标明预留。
- **FR-040**：Platform Admin 登录后 SHOULD 能看到平台管理菜单入口，至少覆盖平台控制台、租户管理、平台用户管理和平台审计预留入口。
- **FR-041**：登录后菜单和入口展示 MUST 根据平台级角色与租户级角色区分；Platform Admin 可见平台管理入口，Tenant Admin 不应看到可执行的平台租户管理入口。
- **FR-042**：系统 MUST 保持一套租户登录和基础布局能力，不应为不同租户复制多套登录页或重复业务逻辑。
- **FR-043**：新增规格、派生文档、说明文字和必要注释 MUST 使用简体中文；代码标识符、路径、命令、接口路径、JSON 字段、表名、类型名、第三方库和算法名除外。

### 关键实体

- **平台管理员 Platform Admin**：拥有平台级 `PLATFORM_ADMIN` 角色的用户，负责租户生命周期、租户用户关系和租户管理员分配；不拥有跨租户文件明文访问权。
- **租户管理员 Tenant Admin**：拥有某个租户下 `TENANT_ADMIN` 角色的用户，只能管理当前租户内部用户、角色、属性、审计等租户内资源。
- **数据拥有者 DO**：某个租户内的数据拥有者身份，用于后续文件上传、策略创建、文件加密和自有文件管理。
- **数据使用者 DU**：某个租户内的数据使用者身份，用于后续查看属性、下载密文和尝试解密。
- **租户 Tenant**：平台下的组织边界，包含名称、编码、描述、状态等信息。当前演示租户包括四川师范大学 `scnu`、深信服科技 `sangfor`、香港友邦保险 `aia-hk`。
- **租户用户关系 TenantUser**：用户与租户之间的成员关系，决定用户是否能进入租户上下文。
- **角色 Role**：平台级或租户级角色定义，至少包含 `PLATFORM_ADMIN`、`TENANT_ADMIN`、`DO`、`DU`。
- **用户角色关系 UserRole**：用户在平台或某个租户下的角色分配。平台级关系不绑定租户，租户级关系必须绑定租户。
- **平台控制台摘要 PlatformDashboardSummary**：平台治理统计信息，可包含租户数量、启用数量、禁用数量、用户数量、成员关系数量和 Tenant Admin 数量。
- **平台操作日志 PlatformAuditLog**：平台级治理操作的审计记录，本阶段可预留，后续补全。

### 宪章对齐

- **混合加密**：本功能不实现文件内容加密或数据密钥 DEK 封装；后续文件加密仍必须遵守“文件内容使用 AES-GCM，DEK 使用 RSA 或 CP-ABE 封装”的边界。
- **真实 CP-ABE**：本功能不实现 CP-ABE 加解密，也不得用平台角色、租户角色或成员关系冒充 CP-ABE 访问策略满足结果。
- **模块边界**：平台租户管理属于平台治理和租户管理能力；不得把密码算法逻辑散落到平台、租户、控制器或处理器中。
- **RSA 与 CP-ABE 对比**：本功能不产生任何算法性能结论，不涉及 Benchmark 对比。
- **策略表达和可解释性**：本功能必须清楚解释平台角色、租户角色和 CP-ABE 属性访问控制之间的边界，避免把 Platform Admin 误解为可解密所有文件。
- **语言规范**：本规格及后续 SpecKit 文档必须使用简体中文。

### 非目标范围

- 不实现 CP-ABE 加密、解密、密钥生成、用户私钥分发或访问树 / LSSS 可视化。
- 不实现文件上传、文件下载、策略加密、解密记录或文件明文查看。
- 不实现完整租户内 RBAC 菜单权限体系，只明确平台级与租户级职责边界。
- 不实现用户撤销、多授权机构、策略隐藏、区块链审计等高级 CP-ABE 扩展能力。
- 不允许 Platform Admin 查看主密钥明文、用户私钥明文、密码明文或绕过 CP-ABE 策略访问明文文件。
- 不将平台管理权限扩展为租户文件内容访问权限。

## 成功标准

### 可衡量结果

- **SC-001**：验收人员可以在 5 分钟内为一个用户确认或分配 `PLATFORM_ADMIN`，并使用该用户进入平台管理入口。
- **SC-002**：Platform Admin 对平台租户管理能力的访问成功率达到 100%，非 Platform Admin 对同类能力的访问拒绝率达到 100%。
- **SC-003**：租户创建流程对空名称、空编码、非法编码和重复编码的拒绝率达到 100%，并返回明确错误。
- **SC-004**：租户被禁用后，Tenant Admin、DO、DU 进入该租户工作台的拒绝率达到 100%，Platform Admin 查看禁用租户信息的成功率达到 100%。
- **SC-005**：用户加入、移出租户后的租户访问结果与成员关系状态 100% 一致。
- **SC-006**：Tenant Admin 分配和移除后，用户在目标租户内的管理员身份变化 100% 可验证，且不影响其他租户。
- **SC-007**：Tenant Admin、DO、DU 均无法创建、启用、禁用或管理平台租户，相关越权测试拒绝率达到 100%。
- **SC-008**：平台管理响应中主密钥明文、用户私钥明文、密码明文和文件明文内容泄露数量为 0。
- **SC-009**：验收文档中能够清楚区分 Platform Admin、Tenant Admin、DO、DU 与 CP-ABE 解密授权边界，不出现“平台管理员可直接解密所有租户文件”的描述。
- **SC-010**：新增规格、计划、任务和说明文档中的业务说明 100% 使用简体中文。

## 假设

- 当前用户注册、登录、多租户基础能力已经存在，本功能在其基础上补齐平台级治理能力。
- 当前演示租户 `scnu`、`sangfor`、`aia-hk` 会继续作为平台租户管理的示例数据存在。
- 第一位 Platform Admin 可以通过初始化数据、后台管理工具或受控运维流程分配；本规格关注平台管理员能力本身，不强制规定首个管理员的创建入口。
- 平台控制台统计和平台审计日志是本阶段可选能力；如果实现成本影响主链路，可先保留入口和设计说明，在后续计划中拆分任务。
- 移除最后一个 Tenant Admin 默认应被保护，避免租户失去管理入口；如后续计划决定允许移除，必须提供明确恢复机制。
- 租户级角色不会替代 CP-ABE 属性和访问策略判断；后续任何解密能力仍必须依赖真实 CP-ABE 或 RSA/AES 混合加密设计。
