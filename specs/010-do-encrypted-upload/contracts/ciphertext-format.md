# 密文容器格式 `GCPABE01`

## 目标

该格式用于本期 AES-256-GCM 认证分块文件密文。它负责有界内存处理、版本识别、关键业务元数据认证、截断、重排和篡改检测。它不包含明文 DEK或 RSA 私钥，也不把 RSA 字段写入通用固定头。每个文件对应的 RSA 密钥信封由服务端作为同一密文包的配套材料返回；可见用户可以下载这些受保护信封，但只有客户端本地持有对应私钥时才能恢复 DEK。

## 二进制布局

```text
偏移      长度             内容
0         8                ASCII 魔数 "GCPABE01"
8         2                大端无符号格式版本，首期为 1
10        4                大端无符号头长度 H
14        H                UTF-8 规范化 JSON 头
随后      重复 chunk_count 次认证分块记录

每个认证分块记录：
0         4                大端无符号块序号 i
4         4                大端无符号明文块长度 P
8         P+16             AES-GCM 密文和 16 字节认证标签
```

读取方必须恰好读取 `chunk_count` 个连续记录；块序号必须从 0 递增，所有 `P` 之和必须等于 `plaintext_size`，非末块的 `P` 必须等于 `chunk_size`。任何额外字节、缺失记录、重复/跳跃序号或长度不一致均视为格式错误。

## 规范化 JSON 头

字段顺序固定，编码使用 UTF-8、无 BOM、无多余空白；数字使用十进制整数，禁止浮点和科学记数法。

```json
{
  "aad_version": "1",
  "algorithm_code": "RSA-OAEP-SHA256",
  "algorithm_version": "1",
  "attempt_id": "uuid",
  "authorization_snapshot_sha256": "64位小写十六进制",
  "content_algorithm": "AES-256-GCM",
  "chunk_count": 1,
  "chunk_size": 4194304,
  "file_id": "uuid",
  "nonce_prefix_base64": "8字节随机 nonce 前缀的 Base64",
  "owner_user_id": 1,
  "plaintext_size": 1024,
  "task_id": "uuid",
  "tenant_id": 1
}
```

字段集合和顺序属于 `aad_version=1` 协议。未来新增字段必须提升 AAD 或容器版本，不能静默改变已有字节表示。

## 密码学参数

- DEK：每次执行由密码学安全随机源生成 32 字节。
- nonce 前缀：每次执行独立生成 8 字节；第 `i` 块 nonce 为该前缀拼接 4 字节大端无符号块序号。一个 DEK下最多允许 `2^32` 个块，且不得重用前缀。
- 算法：`aes-256-gcm`。
- tag：每块 16 字节。
- 头摘要：`SHA-256(规范化 JSON 头字节)`。
- 分块 AAD：`头摘要 || uint32_be(i) || uint32_be(chunk_count) || uint32_be(P)`。
- 上下文摘要：头摘要，供完成接口和 `CryptoEngine` 共同引用。
- 容器摘要：`SHA-256(完整容器字节)`，供上传校验和存储巡检。

## 流式写入顺序

1. 创建任务和执行，获得稳定 ID及服务端验证后的授权快照摘要。
2. 生成 DEK和 nonce，构造规范化头。
3. 写入前缀与头，计算头摘要。
4. 以不超过 `chunk_size` 的 Buffer 读取第 `i` 块，构造唯一 nonce 和分块 AAD，通过 Go `cipher.AEAD.Seal` 得到“密文 + 16 字节 tag”。
5. 写入块序号、明文块长度和认证密文，覆盖已消费的明文 Buffer，再处理下一块。
6. 完成容器 SHA-256，验证块数和总长度，重新读取文件状态确认源文件未变化。
7. 由 Go `CryptoEngine` 的 `RSAEngine` 保护同一个 DEK，随后尽最大可能覆盖 DEK Buffer。

每个记录是独立认证密码分块，但复用同一 DEK；nonce 必须按前缀和序号唯一生成，不得使用相同 nonce 重复初始化 GCM。读取方必须验证完整块序列，不能只验证单个块后输出“完整文件成功”。

## RSA 首个适配器绑定

RSA 适配器使用：

```text
RSA-OAEP
哈希 = SHA-256
MGF1 哈希 = SHA-256
label = SHA-256(AAD)
输入 = 32 字节 DEK
```

RSA 公钥版本、接收者和指纹位于服务端授权快照与 RSA 绑定记录；它们通过 `authorization_snapshot_sha256` 间接进入分块 AAD。下一期应优先由真实 Go `TKN20Engine` 产生新的受保护密钥格式和专属绑定；只有完成不适用评估后才可选择真实 Go BSW07 或其他方案。新引擎复用同一容器布局及上下文摘要规则。

密文下载与解密能力不绑定。服务端只校验文件可见性、租户和对象状态，然后返回密文及密钥信封集合；服务端不保存私钥、原始 AES 密钥，不执行 RSA 解封装、AES 解密或明文返回。Electron 主进程按信封 `key_id` 或公钥指纹从本地安全存储匹配私钥，找不到匹配项或校验失败时必须保留原始密文包。

## 校验要求

- 魔数、版本、头长度、字段集合、字段类型、总长度必须严格校验。
- 解密验证必须按序重建每块 nonce 和 AAD，通过 `cipher.AEAD.Open` 验证；只有全部块、总数和总长度均通过后才可提交可用明文结果。
- 文件 ID、任务 ID、执行 ID、租户、所有者、算法版本和授权摘要必须与服务端记录一致。
- 上传接口必须计算完整容器摘要，不信任客户端 header。
- 日志只记录格式版本、大小、摘要前缀和稳定错误码，不记录 nonce 以外的随机秘密、完整受保护 DEK或本地路径。
