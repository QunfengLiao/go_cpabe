# 研究记录：DO 文件加密上传基础闭环与通用混合加密框架

## 研究范围

本研究解决规格中的技术未知项，并以项目现有 Go/Gin/Gorm、Electron 主进程、RBAC、显式 SQL 迁移和本地存储能力为基础。所有结论服务于工程演示，不构成生产级密码学安全承诺。

## 决策 1：RSA+AES 是首个实现，框架不依赖 RSA

**决定**：定义算法无关的 `DekProtectionAdapter`。RSA-OAEP-SHA-256 是首个注册并在本期唯一启用的适配器。通用协调器与通用数据库表不得包含固定 RSA 接收者、公钥版本或 RSA 密文结构；RSA 专属关联放入独立绑定实体。

**理由**：这同时满足“本期必须有真实可运行 RSA+AES 闭环”和“下一期接入 BSW07 时不复制文件流程”。适配器边界只暴露能力描述、授权校验、DEK 保护结果和版本化专属绑定。

**备选方案**：

- 在文件 Service 中按 `if algorithm == RSA` 分支：短期简单，但会把下一期算法、授权表单和数据字段绑死在上传流程中，拒绝。
- 先实现仅含 RSA 字段的数据表，后续再重构：会造成历史迁移和安全审计重复，拒绝。

## 决策 2：文件密码学操作位于客户端设备上的本地 Go Crypto Worker

**决定**：渲染进程负责展示和收集非敏感授权选择；Electron 主进程负责文件对话框、受控 IPC、本地 Go Crypto Worker 生命周期、临时密文和上传。Crypto Worker 使用 `backend/internal/crypto` 的统一 `CryptoEngine` 执行 DEK 生成、AES、RSA、哈希及文件流，且不返回明文 DEK。渲染进程不获得文件完整路径、DEK、私钥或临时密文路径。

**理由**：该边界满足项目宪章“后端统一 `CryptoEngine`”和“真实 CP-ABE 必须使用 Go 库”的约束，同时保持明文文件与 DEK 不进入远程服务端。Electron 仍遵循上下文隔离、单用途 IPC 和发送方校验；主进程只启动固定路径且经过完整性校验的本地二进制，不经过 shell。[Electron 安全指南](https://www.electronjs.org/docs/latest/tutorial/security)、[Electron 上下文隔离](https://www.electronjs.org/docs/latest/tutorial/context-isolation)、[Go `os/exec`](https://pkg.go.dev/os/exec)

**备选方案**：

- 在 React 渲染线程或 Node 工作线程实现密码算法：无法复用宪章要求的真实 Go CP-ABE 和后端 `CryptoEngine`，且扩大 Electron 密码学实现面，拒绝。
- 上传明文给 Go 服务端加密：违反明文文件默认只在本地的安全边界，拒绝。
- 在 Electron 主线程直接同步加密：可能阻塞窗口和 IPC，拒绝。

## 决策 3：使用有界内存的认证分块 AES-256-GCM

**决定**：使用 32 字节随机 DEK、8 字节随机 nonce 前缀、4 MiB 默认分块和每块 16 字节认证标签。第 `i` 块 nonce 为“随机前缀 + 4 字节大端 `i`”；每块 AAD 绑定规范化头摘要、块序号、总块数和明文块长度。读取方必须验证连续块序号、块数和总长度。

**理由**：Go 标准 `cipher.AEAD.Seal` 接收一个完整消息切片，直接对 1 GiB 文件调用会违反内存目标。固定上限分块可将峰值内存限制在分块大小附近；每块 nonce 对同一 DEK 唯一，块级 AAD 与容器总量约束共同检测篡改、截断、重复、缺失和重排。[Go `crypto/cipher`](https://pkg.go.dev/crypto/cipher)

**备选方案**：

- 单消息 GCM：Go 标准 AEAD 没有增量 `Seal` API，整文件载入内存不满足 1 GiB 和内存目标，拒绝。
- 每块使用独立随机 nonce：需要保存大量 nonce 且增加碰撞和格式复杂度；改用随机前缀加单调计数器。
- AES-CTR + HMAC：可行但需要双密钥和组合协议，宪章已指定 AES-GCM，拒绝。

## 决策 4：密文容器与业务上下文绑定

**决定**：定义 `GCPABE01` 版本化容器。头部采用确定字段顺序的 UTF-8 JSON，包含格式版本、任务/执行/租户/所有者、原始大小、分块大小、总块数、nonce 前缀、内容算法、DEK 保护算法版本和授权快照摘要。每块 AAD 以头摘要为基础并加入块序号、总块数和块长度；头 SHA-256 摘要同时作为 RSA-OAEP label。容器整体另计算 SHA-256。

**理由**：AAD 使元数据替换导致 GCM 验证失败；OAEP label 使受保护 DEK不能在不同任务、接收者或公钥版本之间替换；整体哈希用于上传传输和存储对象一致性校验。三者分别承担密码学绑定、密钥绑定和工程完整性职责。

**备选方案**：

- 只保存 nonce 和 tag，不绑定业务字段：无法阻止合法密文与错误任务元数据调包，拒绝。
- 把受保护 DEK写入 AES 头并一并认证：会形成生成顺序耦合，且未来算法密文头大小差异大；选择独立受保护密钥记录和共同上下文摘要。

## 决策 5：RSA-OAEP 显式使用 SHA-256 和上下文 label

**决定**：首个 `RSAEngine` 使用 RSA 3072 位公钥、OAEP 填充、Go `rsa.EncryptOAEP(sha256.New(), rand.Reader, ...)` 和上下文摘要 label。加解密必须使用相同 SHA-256 与 label，不使用 PKCS#1 v1.5。

**理由**：Go 标准库直接支持以 SHA-256、密码学安全随机源和任意上下文 label 执行 RSA-OAEP，符合宪章 Go 密码模块边界。[Go `crypto/rsa`](https://pkg.go.dev/crypto/rsa)

**备选方案**：

- 默认 OAEP 参数：会隐式使用 SHA-1且协议不明确，拒绝。
- PKCS#1 v1.5：不符合项目 RSA-OAEP 约束，拒绝。
- RSA 直接加密文件：违反混合加密宪章，拒绝。

## 决策 6：客户端生成并保护私钥，服务端只保存公钥版本

**决定**：本地 Go Crypto Worker 生成 RSA 3072 位密钥对；公钥以 SPKI PEM 上传。PKCS#8 PEM 私钥通过单用途本地进程响应进入 Electron 主进程后，立即使用 Electron 33 支持的同步 `safeStorage.encryptString` 保护并存入按账号和租户隔离的本地文件，再覆盖临时 Buffer。该同步调用只处理小体积密钥且不进入高频循环。安全存储不可用，或 Linux 后端为 `basic_text`/`unknown` 时，禁止保存密钥并给出明确提示。

**理由**：现有项目已使用 `safeStorage` 保护账号凭据和刷新会话，可以复用相同平台边界。Electron 文档说明其依赖系统提供的密码能力，同时不同系统安全语义不同，因此必须检测可用性和后端。[Electron safeStorage](https://www.electronjs.org/docs/latest/api/safe-storage)

**备选方案**：

- 服务端生成并返回私钥：扩大私钥暴露面并与本期安全假设冲突，拒绝。
- 私钥明文写入文件或 `localStorage`：违反规格，拒绝。
- 不实现密钥生成，仅依赖预置演示公钥：无法形成可维护的公钥版本能力，拒绝。

## 决策 7：首期使用受控本地密文存储，不复用头像静态 URL

**决定**：扩展存储抽象，新增密文暂存、提交、打开下载流、删除和巡检能力。首期实现本地受控目录，路径由 `tenant_id/task_id/random_id` 派生且仅作为内部对象键。密文不注册 Gin 静态路由，下载必须经过鉴权 Handler。

**理由**：仓库已有本地头像存储，但头像通过公开静态前缀访问，不适合密文。独立密文接口可在后续无业务改动地替换为 MinIO/S3。

**备选方案**：

- 直接复用 `SaveAvatar`：对象命名、公开 URL和权限模型不符合密文要求，拒绝。
- 本期同时引入 MinIO：增加部署依赖且不是形成闭环的必要条件，拒绝。

## 决策 8：任务采用幂等键、执行记录和行锁状态机

**决定**：任务拥有不可变授权快照和唯一幂等键；每次执行/重试单独记录。状态变化在事务中对任务和当前执行执行锁定读，验证期望版本和合法转换后更新。完成接口可重复调用并返回相同完成结果。

**理由**：MySQL 官方说明在读取后更新关联数据时，普通一致性读不足以防止并发修改，`SELECT ... FOR UPDATE` 可锁定目标记录。唯一约束负责请求去重，行锁和版本负责取消、完成、重试竞争。[MySQL InnoDB 锁定读](https://dev.mysql.com/doc/refman/8.4/en/innodb-locking-reads.html)

**备选方案**：

- 仅在应用内检查状态：多进程或并发请求可同时通过，拒绝。
- Redis 分布式锁作为唯一一致性机制：数据库仍是事实源，锁过期会引入双写风险；Redis 仅用于限流与并发计数。

## 决策 9：上传完成采用暂存对象与补偿，不声称跨资源事务

**决定**：密文先写入不可下载的暂存对象，服务端边写边计算 SHA-256并校验长度。完成请求在数据库事务内保存文件、密文、受保护密钥、算法绑定、Benchmark 与审计，再将对象记录置为可用。事务失败后删除暂存对象；删除失败登记孤儿对象并由清理命令重试。

**理由**：文件系统和 MySQL 之间不存在原子提交。显式补偿能保证“数据库完成但对象不存在”和“对象存在但无有效元数据”均可识别处理。

**备选方案**：

- 上传后立即公开对象：数据库失败会产生可下载孤儿，拒绝。
- 把 1 GiB 密文存入 MySQL BLOB：影响数据库备份和吞吐，且不利于后续对象存储替换，拒绝。

## 决策 10：重试总是重新加密，不做断点续传

**决定**：网络上传失败、应用退出或可恢复服务端错误产生新执行；重新选择/校验源文件、生成新 DEK和 nonce、重新输出密文并上传。首期不复用旧临时密文、不续传、不暂停。

**理由**：这使 DEK 生命周期、临时文件恢复和上传偏移校验保持简单且可审计，符合规格默认假设。1 GiB 上限使重新执行在演示范围内可接受。

**备选方案**：

- 保留密文与会话做断点续传：需要安全持久化更多恢复状态、上传分片协议和过期清理，延至后续。

## 决策 11：真实审计落库，Redis 只做短期控制

**决定**：替换本功能路径上的 `NoopAuditRecorder`，并在创建、校验、授权快照、公钥选择、上传、完成、失败、取消、重试、下载和清理补偿调用点显式写入 MySQL 审计表。客户端报告的 AES、DEK 保护和 Benchmark 事件标为 `CLIENT_REPORTED`，服务端确认事件标为 `SERVER_OBSERVED`。Redis 使用原子令牌桶限制用户/租户请求频率，使用带 TTL 租约控制并发；终态释放租约，巡检修复过期租约。Redis 故障默认安全失败，可配置回退到数据库非终态计数，不把 Redis 当事实源。

**理由**：规格要求安全事件可追踪；空实现不满足验收。审计数据需要持久一致，限流计数允许短期失效。

## 决策 12：文件选择和源文件变化检测

**决定**：通过主进程 `dialog.showOpenDialog` 选择单文件，主进程保存随机内存句柄；工作线程打开文件描述符，读取前后比较文件标识、大小和高精度修改时间，并在流中计算明文摘要供本地诊断但不上传该摘要。任何变化使执行失败并删除临时密文。

**理由**：Electron 原生文件对话框属于主进程能力；Node 文件句柄和流支持有界内存读取。[Electron dialog](https://www.electronjs.org/docs/latest/api/dialog)、[Node.js 文件系统](https://nodejs.org/api/fs.html)

**备选方案**：

- 渲染进程直接提交本地绝对路径：扩大 XSS 后的文件读取能力，拒绝。
- 仅检查扩展名：不能证明文件可读或期间未变化，拒绝。

## 决策 13：下一期 CP-ABE 选择遵循 TKN20 优先门禁

**决定**：本期不实现任何 CP-ABE。下一期必须先对 Cloudflare CIRCL TKN20 的真实 Go 实现进行 API、维护状态、性能、策略表达和桌面本地 Worker 适配评估；评估通过则实现 `TKN20Engine`。只有形成可复核的不适用结论后，才可选择具有真实 Go 实现的 BSW07 或其他方案。

**理由**：项目宪章明确要求优先评估 TKN20，并禁止数学模拟或手写示例冒充真实 CP-ABE。把下一期预先锁定为 BSW07 会绕过该门禁。

**备选方案**：

- 直接按原功能名称实现 BSW07：缺少 TKN20 不适用证据，违反宪章优先级，拒绝。
- 在本期实现测试版 CP-ABE：扩大范围并可能形成伪实现，拒绝。

## 已解决的规格待确认项

| 原问题 | 本计划结论 |
|--------|------------|
| RSA 能力现状 | 当前仓库无 RSA 密钥实体；本期新增客户端生成、公钥登记、版本和禁用能力。 |
| 私钥位置 | 仅客户端系统安全存储保护，不进服务端。 |
| RSA 解密页面 | 不在本期；通过自动化往返测试验证。 |
| 密文存储 | 首期受控本地存储，经接口可替换对象存储。 |
| 文件上限 | 1 GiB。 |
| 文件类型 | 任意二进制；类型仅展示。 |
| 断点续传 | 不支持。 |
| 暂停与恢复 | 不支持；支持可安全取消和全量重试。 |
| Electron 退出恢复 | 只恢复任务标识并标记中断，不持久化 DEK 或可继续加密状态。 |
| AES 大文件格式 | 4 MiB 认证分块 AES-256-GCM；8 字节随机前缀加 4 字节块序号形成唯一 nonce，每块 16 字节 tag。 |
| RSA 接收者数量 | 每个文件一个接收者。 |
| 历史私钥 | 本机按公钥指纹保留；跨设备备份不在本期。 |
| 用户移出租户 | 禁止新操作；历史关系保留审计，本期不提供解密。 |
| DO 删除密文 | 本期不支持删除已完成密文。 |
| 本地密文副本 | 上传完成或失败补偿后删除应用临时副本。 |
| 加密执行位置 | Electron 启动的本地 Go Crypto Worker；远程服务端不接收明文或 DEK。 |
| 访问策略页面 | 本期不接入上传页面。 |
