# 快速验证：多接收者文件中心与本地解密闭环

## 前置条件

- 后端 MySQL、Redis 和 HTTP 服务可用。
- 桌面端能启动 Electron 主进程和本地 Crypto Worker。
- 同一租户内至少有 5 个成员：1 个 DO、3 个目标 DU、1 个普通租户成员。
- DO 和 3 个 DU 已登记有效 RSA 公钥；普通租户成员可有或没有 RSA 公钥。

## 场景一：一个文件分享给三个人

1. 使用 DO 登录并进入“数据加密”。
2. 选择一个本地测试文件。
3. 在接收者列表中选择 3 个 DU；确认 DO 自己以“文件拥有者”身份锁定勾选。
4. 点击“开始加密并上传”。
5. 预期结果：
   - 进度经历校验、AES-GCM、DEK 保护、上传、提交元数据和完成。
   - 文件中心“我的加密文件”出现该文件。
   - 详情显示 4 名接收者，且不是数字 ID。
   - 数据库中该文件只有 1 个可用密文对象，且有 4 条 protected DEK。

## 场景二：普通租户成员下载全部密文

1. 使用普通租户成员登录同一租户。
2. 进入“文件中心 > 全部密文”。
3. 找到场景一创建的文件。
4. 预期结果：
   - 列表不显示分享关系、可解密或未授权状态。
   - 即使没有 `file.read`，仍可以查看非敏感密文元数据、下载原始密文和获取完整密钥信封。
   - 后端返回 200 和密文；能否恢复明文只由 Electron 本地私钥决定。

## 场景三：统一下载并在本地解密

1. 使用任一授权 DU 登录。
2. 进入“文件中心 > 全部密文”。
3. 点击统一“下载”按钮。
4. 在系统原生窗口中选择输出目录。
5. 若目录已有同名文件，再次执行解密。
6. 预期结果：
   - 主进程下载原始 `.enc` 密文，逐个尝试本地 RSA-OAEP 信封，再执行 AES-GCM 解密。
   - 第一次解密成功并打开所在文件夹，展示 RSA 解封耗时、AES 解密耗时和完整性校验通过。
   - 第二次生成“文件名 (1).扩展名”等不重名文件。
   - 不自动运行或直接打开明文文件。

## 场景四：真实进度和取消

1. 选择较大文件执行加密。
2. 观察 AES 阶段字节数增长。
3. 观察 RSA 阶段接收者进度。
4. 观察上传阶段上传字节增长。
5. 在元数据提交前取消。
6. 预期结果：
   - 进度不随机跳动。
   - 取消后任务不进入 AVAILABLE。
   - 临时密文和未完成上传被清理或进入补偿清理。

## 场景五：性能指标

1. 完成一次多接收者加密。
2. 打开“我的加密文件”和详情抽屉。
3. 预期结果：
   - 列表显示本地加密耗时摘要。
   - 详情显示 AES、RSA DEK 保护总耗时、平均/最小/最大单接收者耗时、上传、元数据提交和总耗时。
   - 接收者详情显示每个接收者 DEK 保护耗时。

## 场景六：跨租户攻击

1. 使用租户 A 的 DO 尝试选择租户 B 的接收者或公钥。
2. 使用租户 B 用户访问租户 A 文件 ID、密文下载地址和解密材料地址。
3. 预期结果：
   - 创建任务时拒绝跨租户接收者和公钥。
   - 文件详情、密文下载和解密材料接口均拒绝跨租户访问。

## 场景七：账号切换

1. 使用授权用户 A 打开文件中心并加载解密材料相关页面状态。
2. 切换到用户 B 或其他租户。
3. 预期结果：
   - 文件列表、接收者、公钥和本地私钥运行时缓存被清空。
   - 用户 B 不能使用用户 A 的权限或私钥解密文件。

## 场景八：错误私钥与认证失败

1. 使用只有错误 RSA 私钥的租户成员下载场景一文件，确认后端返回 200 和原始密文。
2. 预期客户端不提示“无权限”或“未授权”，而是保存 `文件名.enc`，允许稍后导入其他私钥重试。
3. 篡改密文或 Tag 后重新下载，预期 AES-GCM 完整性校验失败，不输出任何明文，原始密文仍被保留。

## 场景九：服务端密文边界

1. 抓取上传请求和服务端日志/数据库内容。
2. 预期网络中只有密文流、Nonce/Tag/算法元数据、哈希和 RSA 密钥信封；不存在明文文件字节或原始 AES DEK。
3. 删除操作由 RBAC 拒绝普通成员，证明 RBAC 仍控制管理操作，但不控制密文下载和本地明文恢复。

## 场景十：审计 Outbox 故障恢复

1. 完成 `014_audit_outbox.sql` 迁移，并启动后端但暂不运行审计 Dispatcher。
2. 执行一次多接收者任务创建和完成操作。
3. 查询 `audit_outbox`，确认安全关键事件处于 `PENDING`，事件中不存在 Token、路径、私钥、DEK、完整受保护密钥或原始错误。
4. 同时启动两个 Dispatcher，或分别运行两次单批命令。
5. 预期结果：
   - 每个 `event_public_id` 在 `audit_logs` 中最多出现一次。
   - outbox 最终进入 `DELIVERED`，正式日志时间使用 `occurred_at`，不是重放时间。
   - 模拟“正式日志已写入但 outbox 状态更新失败”后再次执行，仍收敛为一条正式日志。
6. 注入可重试写入错误，确认事件按指数退避进入 `RETRY`；超过最大次数进入 `DEAD_LETTER`。
7. 让 Dispatcher 在领取后退出，等待租约过期并再次运行，确认新 `lock_token` 能重新领取，旧 Worker 不能覆盖结果。
8. 断开整个 MySQL 后触发无业务写事务审计，确认普通日志只包含事件 UUID、动作和稳定错误分类，不包含 Metadata 或原始数据库错误。

## 建议验证命令

```powershell
# 后端
go test ./internal/domain ./internal/repository ./internal/service ./internal/handler ./internal/crypto
go test ./internal/migrations ./internal/repository ./internal/service ./cmd/audit-dispatcher
go vet ./...

# 桌面端
cd desktop
npm.cmd test -- --run
npm.cmd run typecheck
npm.cmd run build
```

如果当前 shell 找不到 `go.exe`，需要使用 GoLand 配置的 Go SDK 或将 Go SDK 加入 PATH 后再执行后端测试。
