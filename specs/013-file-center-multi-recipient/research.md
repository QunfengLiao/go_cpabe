# 研究结论：多接收者文件中心与本地解密闭环

## 决策 1：复用 protected_keys 和 rsa_protected_key_bindings 表表达一对多

**决定**：保留通用 `protected_keys` 表和 RSA 专属 `rsa_protected_key_bindings` 表，移除或替换阻碍多接收者的唯一约束：`protected_keys.file_id`、`protected_keys.task_attempt_id` 不再单独唯一；RSA 绑定增加同一文件、接收者、公钥版本的唯一约束语义。

**理由**：现有模型已经把通用受保护 DEK 和 RSA 专属绑定分离，符合算法可插拔边界。调整约束即可支持多接收者，避免新建重复表或在文件表中拼接接收者。

**替代方案**：
- 新建 `file_recipients` 表：会与 `rsa_protected_key_bindings` 重叠，增加同步复杂度。
- 在 `encrypted_files` 增加 `recipient_ids` 字符串：无法安全校验公钥版本、protected DEK 和单接收者授权，已被规格禁止。

## 决策 2：加密任务创建阶段冻结接收者授权快照

**决定**：创建任务时后端接受 `recipients[]`，逐条校验用户、租户、公钥状态和版本，并把规范化接收者数组写入任务授权快照。完成任务时后端按快照校验提交的每份 protected DEK。

**理由**：加密任务可能跨越较长时间，公钥状态和用户列表可能变化。冻结快照可以保证任务开始时的授权意图与完成时入库事实一致，同时历史公钥版本可继续解密。

**替代方案**：
- 完成任务时重新使用前端提交的接收者：会信任客户端，容易被替换接收者或公钥。
- 每次查询实时取当前 ACTIVE 公钥：会破坏历史文件对旧公钥版本的绑定。

## 决策 3：本地 worker 一次 AES 加密，多次 RSA 保护同一个 DEK

**决定**：Crypto Worker 在一次加密请求中生成一个 DEK、写出一份 GCPABE01 密文容器，然后遍历接收者授权参数，为每个接收者执行 RSA-OAEP-SHA256 并返回 protected key 数组及单接收者耗时。

**理由**：这是混合加密多接收者的正确模型，能避免重复读取和重复上传大文件，并提供公平的 RSA 接收者数量性能指标。

**替代方案**：
- 主进程多次调用 worker 对同一文件加密：违反“一份密文”要求。
- 由后端执行 RSA-OAEP：会把明文 DEK 传给后端，违反安全边界。

## 决策 4：文件中心使用统一 scope 查询

**决定**：提供一个文件中心查询能力，scope 为 `tenant_cloud`、`owned_by_me`。全部密文按当前租户有效成员读取，后端不计算解密授权状态，前端不呈现“分享给我”或可解密推断。

**理由**：三类列表展示字段高度一致，差异主要是过滤范围和授权状态。统一查询能减少重复逻辑，并让企业云盘“可见不等于可解密”的语义更清晰。

**替代方案**：
- 保持三个完全独立接口：会造成详情、接收者摘要、权限按钮和状态展示重复实现。

## 决策 5：解密交互使用输出目录而不是保存文件对话框

**决定**：主进程通过原生文件夹选择获取输出目录，使用安全文件名和自动编号生成最终文件路径。解密成功后调用安全能力打开所在文件夹，但不直接运行或打开明文文件。

**理由**：目录选择符合用户要求，并可统一处理重名、非法文件名和安全打开。保存文件对话框容易诱导覆盖目标文件，当前实现存在删除目标文件再重命名的静默覆盖风险。

**替代方案**：
- 继续使用保存文件对话框：与需求不一致，且覆盖语义更难统一。

## 决策 6：性能指标分为任务级与接收者级

**决定**：任务级记录验证、AES、DEK 保护总耗时、上传、元数据提交、总耗时、文件大小、密文大小、接收者数量、protected DEK 总大小和执行结果；接收者级记录每个 RSA-OAEP 耗时、公钥版本和 protected DEK 大小。

**理由**：后续 RSA、BSW07、Waters11、TKN20 等算法比较需要区分文件加密、DEK 封装和网络上传。接收者级耗时支持分析 RSA 随接收者数量增长的成本。

**替代方案**：
- 只记录总耗时：无法公平比较算法。
- 只记录平均耗时：无法排查单接收者异常和最大耗时。

## 决策 7：安全关键审计采用独立事务 Outbox 和幂等 Dispatcher

**决定**：新增独立 `audit_outbox` 表。安全关键数据库写操作在同一 MySQL 事务内写业务事实和经过白名单过滤的 outbox 事件；独立 Dispatcher 使用 `SKIP LOCKED`、处理租约、指数退避和死信状态，将事件按稳定 UUID 幂等写入 `audit_logs`。拒绝、下载、进度等无业务写事务事件使用短事务入队，数据库整体不可用时只输出不含元数据的结构化告警。

**理由**：业务表与 `audit_logs` 直接双写会出现业务成功但审计丢失，或者审计成功但业务回滚。事务 outbox 把“业务事实存在”和“审计事件待投递”绑定在同一个数据库提交中；Dispatcher 的幂等键和租约使进程崩溃、重复领取和并发消费者不会制造重复正式日志。入队前过滤可以避免 outbox 成为密钥、Token 或本地路径泄漏通道。

**替代方案**：
- 直接尽力写 `audit_logs` 并打印错误：无法证明安全关键事件可恢复，只适合作为数据库整体不可用时的最后告警。
- 复用 `orphan_storage_objects`：该表的消费者会调用密文删除能力，且记录结构不能重放完整审计事件，存在误清理和静默丢失风险。
- Redis 队列：Redis 不是本项目业务事实源，无法与 MySQL 业务事务原子提交；服务崩溃或网络分区会留下双写窗口。
- 只在 Dispatcher 内使用进程锁：无法覆盖多实例和进程重启，拒绝。

**安全边界**：outbox 只保存正式审计允许的标量元数据，限制 JSON 大小，不保存原始数据库错误、堆栈、私钥、DEK、Token、本地路径或完整受保护密钥；死信记录仍遵守相同边界。
