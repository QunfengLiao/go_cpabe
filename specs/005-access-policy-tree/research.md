# 研究记录：访问策略管理与 DATA_OWNER 可视化访问树构建

## 决策 1：访问树编辑器运行在 Electron Renderer，并以 React Flow 作为画布基础

**选择**：在 `desktop/src/renderer/src` 中实现访问树编辑器，优先使用 React Flow 负责节点、边、拖拽、缩放、controls、minimap 和选择交互。

**理由**：当前桌面端已经是 React + Electron + Vite。React Flow 能直接表达节点和边，适合访问树可视化，比从零实现画布交互更快进入产品化体验。

**备选方案**：

- 自研 SVG/Canvas 画布：可控性高，但拖拽、缩放、选中、边连接、minimap 和可访问性成本较高。
- 只做普通表单：实现快，但不满足“高交互、有产品感”的核心目标，也不利于解释访问结构。

## 决策 2：业务访问树 JSON 与 React Flow nodes/edges 分离

**选择**：保存和后端校验使用 `policy_tree_json`；React Flow 的 `nodes/edges` 只作为 UI 编辑状态，通过 `flowToTree` 和 `treeToFlow` 双向转换。

**理由**：React Flow 节点包含坐标、选中状态和 UI 数据，不适合直接作为后端和后续 CP-ABE 策略输入。业务 JSON 更稳定，可复用到后端校验、表达式生成、文件上传和后续可视化。

**备选方案**：

- 直接保存 React Flow nodes/edges：回显方便，但后端校验树结构更复杂，且会把 UI 实现细节写入业务数据。
- 只保存表达式：表达式便于阅读，但无法稳定回显可视化访问树，也不利于节点级错误定位。

## 决策 3：后端以 policy_tree_json 为权威来源重新生成 policy_expr

**选择**：后端保存访问策略时重新校验 `policy_tree_json`，并生成标准 `policy_expr` 入库；客户端传入的 `policyExpr` 仅作为参考或调试信息。

**理由**：前端可被篡改，表达式和 JSON 可能不一致。后端以 JSON 为权威能保证访问树结构、属性引用和表达式展示保持一致。

**备选方案**：

- 完全信任前端表达式：实现简单，但存在伪造和不一致风险。
- 同时要求表达式和 JSON 严格完全一致：安全性更强，但第一阶段成本更高，且表达式格式化差异容易造成误报。

## 决策 4：前端状态管理使用 hooks + reducer/Context

**选择**：访问树编辑器内部使用 hooks + reducer 或局部 Context 管理状态；暂不引入 Zustand 或 Redux。

**理由**：当前项目没有全局状态库，访问树编辑状态主要局限在编辑器页面。局部 reducer 足够表达复杂状态迁移，同时避免新增全局依赖。

**备选方案**：

- Zustand：轻量但仍是新增架构依赖，适合后续多个页面共享复杂编辑状态时再引入。
- Redux：治理能力强，但对当前规模偏重。

## 决策 5：本地草稿缓存作为第五阶段增强

**选择**：第一至四阶段不把草稿缓存作为阻塞项；第五阶段用 localStorage 保存非敏感草稿，并记录远端更新时间用于冲突提示。

**理由**：核心闭环是策略构建、保存和回显。草稿能改善桌面体验，但容易引入远端数据冲突，应该在真实保存链路稳定后实现。

**备选方案**：

- 一开始使用 IndexedDB：容量和结构更好，但当前数据规模不需要。
- 通过 preload 写本地文件：会扩大 Electron 系统能力暴露面，不适合作为默认方案。

## 决策 6：角色码保留现有后端常量，前端展示做兼容映射

**选择**：后端继续使用当前 `RoleDO`/`RoleDU` 等稳定常量；前端菜单和文案把 `DO` 映射为 DATA_OWNER，把 `DU` 映射为 DATA_VISITOR。

**理由**：需求文本使用 DATA_OWNER/DATA_VISITOR，但当前代码已实现 DO/DU 角色。为了避免本功能重构登录、租户成员和角色分配体系，采用兼容映射更稳妥。

**备选方案**：

- 全量重命名后端角色码：会影响认证、租户角色分配、历史测试和已有数据，不符合非目标。
- 只在文档使用 DATA_OWNER，代码完全忽略：容易造成菜单判断和权限测试歧义。

## 决策 7：访问树校验放入独立 policytree 包

**选择**：后端新增 `backend/internal/pkg/policytree`，包含树结构、校验和表达式生成；service 调用该包，不在 handler 中递归解析。

**理由**：访问树校验是核心业务规则，需要可复用、可测试、可注释。独立包能服务模板校验、访问策略保存和后续文件上传复用。

**备选方案**：

- 放在 service 文件内：初期快，但测试和复用较差。
- 放在 handler 内：违反分层，容易混入请求绑定和业务规则。

## 决策 8：数据库使用 JSON 或 TEXT 由迁移兼容性决定

**选择**：优先使用 MySQL JSON 类型保存 `attr_values` 和 `policy_tree_json`；如果现有环境兼容性受限，则退回 TEXT 并在 service 层保证 JSON 合法性。

**理由**：MySQL 8 支持 JSON 类型，能表达结构数据；但项目演示环境可能需要兼容已有迁移方式，TEXT 回退可以降低部署风险。

**备选方案**：

- 拆成节点表：查询复杂且不利于快速回显，当前阶段没有必要。
- 只保存表达式：无法满足 JSON 预览和可视化回显。
