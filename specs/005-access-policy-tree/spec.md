# 功能规格：访问策略管理与 DATA_OWNER 可视化访问树构建

**功能分支**：`feat/access-policy-tree`

**创建日期**：2026-07-08

**状态**：草案

**输入**：用户描述：“新增访问策略管理与 DATA_OWNER 可视化访问树构建功能，支持 PLATFORM_ADMIN 管理属性字典和策略模板，支持 DATA_OWNER 基于开放属性与模板构建、预览、保存和回显可视化访问树，并按角色与租户边界控制菜单和接口访问。”

## 用户场景与测试（必填）

### 用户故事 1 - DATA_OWNER 可视化构建访问策略（优先级：P1）

作为 DATA_OWNER，我希望在一个可视化访问树编辑器中基于平台开放的属性和模板构建访问策略，并实时看到策略表达式、结构化 JSON 和校验结果，这样我可以在不手写复杂表达式的情况下创建后续文件加密共享可复用的访问控制规则。

**优先级理由**：这是本功能的核心用户价值。没有可视化构建能力，DATA_OWNER 无法独立创建可解释、可回显、可持久化的访问策略。

**独立测试**：使用一组已启用属性字典和一个策略模板，DATA_OWNER 可以独立完成新建访问树、编辑节点、查看表达式与 JSON、触发校验、保存策略并重新打开回显。

**验收场景**：

1. **假设** 当前用户是 DATA_OWNER 且属于目标租户，**当** 用户进入“访问策略构建”页面，**那么** 系统展示可视化访问树编辑器、可选属性字典、可选策略模板、节点配置面板、表达式预览、JSON 预览和校验提示。
2. **假设** DATA_OWNER 已添加一个 OR 根节点、一个 AND 子节点和多个属性叶子节点，**当** 用户修改属性、操作符或属性值，**那么** 系统实时更新访问树 JSON 和策略表达式。
3. **假设** 访问树存在缺少属性值、孤立节点或 AND/OR 子节点不足等问题，**当** 用户保存策略，**那么** 系统阻止保存、给出明确原因，并在画布中高亮问题节点。
4. **假设** DATA_OWNER 已保存一条访问策略，**当** 用户从“我的访问策略”进入编辑，**那么** 系统根据已保存的访问树 JSON 回显原访问树结构、节点配置和预览内容。

---

### 用户故事 2 - PLATFORM_ADMIN 管理策略基础能力（优先级：P2）

作为 PLATFORM_ADMIN，我希望维护系统允许使用的访问策略属性字典和策略模板，这样可以统一约束 DATA_OWNER 能使用哪些属性、哪些模板和哪些基础策略结构。

**优先级理由**：平台开放的属性和模板是 DATA_OWNER 构建策略的来源，直接影响访问策略的一致性、可控性和后续 CP-ABE 策略转换质量。

**独立测试**：PLATFORM_ADMIN 可以独立完成属性字典和策略模板的新增、查看、修改、禁用或删除；DATA_OWNER 只能看到已启用且平台开放的属性和模板。

**验收场景**：

1. **假设** 当前用户是 PLATFORM_ADMIN，**当** 用户进入“访问策略管理”，**那么** 系统允许其维护属性字典和策略模板。
2. **假设** PLATFORM_ADMIN 禁用某个属性，**当** DATA_OWNER 构建或保存访问树，**那么** 该属性不能作为新的可选属性使用，已引用该属性的访问树在校验时提示属性不可用。
3. **假设** PLATFORM_ADMIN 创建策略模板，**当** DATA_OWNER 新建策略时选择该模板，**那么** 系统以模板访问树作为起点，并允许 DATA_OWNER 继续编辑后保存为自己的策略。
4. **假设** PLATFORM_ADMIN 管理策略模板，**当** 平台管理员完成模板维护，**那么** 该行为不会授予其访问租户文件或解密租户文件的权限。

---

### 用户故事 3 - 按角色与租户边界访问策略（优先级：P3）

作为系统安全维护者，我希望菜单和访问策略操作都严格遵守角色、租户和所有者边界，这样不同角色只能看到和操作自己有权访问的策略能力。

**优先级理由**：访问策略会影响后续文件加密共享，权限边界错误会破坏租户隔离和数据拥有者控制权。

**独立测试**：分别以 PLATFORM_ADMIN、DATA_OWNER、TENANT_ADMIN、DATA_VISITOR 身份访问菜单和策略操作，验证展示内容、查看范围、编辑权限和拒绝响应。

**验收场景**：

1. **假设** 当前用户是 PLATFORM_ADMIN，**当** 系统生成菜单，**那么** 用户看到“访问策略管理”，但看不到 DATA_OWNER 的“访问策略构建”入口。
2. **假设** 当前用户是 DATA_OWNER，**当** 系统生成菜单，**那么** 用户看到“访问策略构建”和“我的访问策略”，但看不到 PLATFORM_ADMIN 的“访问策略管理”入口。
3. **假设** 当前用户是 TENANT_ADMIN，**当** 用户查看本租户访问策略，**那么** 系统允许查看列表和详情，但不允许编辑或删除 DATA_OWNER 创建的策略。
4. **假设** 当前用户是 DATA_VISITOR，**当** 用户尝试访问访问策略构建或写操作，**那么** 系统拒绝访问并返回明确的权限错误。
5. **假设** DATA_OWNER 尝试编辑其他 DATA_OWNER 创建的策略，**当** 用户提交修改或删除，**那么** 系统基于 owner_id 边界拒绝操作。
6. **假设** 任意租户角色尝试访问非所属租户策略，**当** 用户请求列表、详情、创建、编辑或删除，**那么** 系统基于 tenant_id 边界拒绝操作。

### 边界情况

- 当访问树根节点为空时，系统必须阻止保存，并提示“访问树不能为空”。
- 当 AND 或 OR 逻辑节点少于两个子节点时，系统必须提示该逻辑节点不满足构建规则。
- 当属性叶子节点没有属性、操作符或属性值时，系统必须提示缺失字段，并高亮对应节点。
- 当属性类型是 enum 时，属性值只能来自平台维护的可选值；当类型是 string 或 number 时，系统必须按类型校验输入。
- 当访问树中出现循环引用、孤立节点、已禁用属性或未开放属性时，系统必须拒绝保存。
- 当前端生成的策略表达式与访问树 JSON 不一致时，系统必须以后端校验通过的访问树 JSON 为准，并生成或返回标准策略表达式。
- 当策略模板被禁用后，DATA_OWNER 不应再将其作为新策略起点；已基于该模板创建的策略仍按自身保存内容展示和校验。
- 当保存、更新或删除失败时，系统必须返回可理解的失败原因，避免只展示笼统失败。

## 需求（必填）

### 功能需求

- **FR-001**：系统必须为 PLATFORM_ADMIN 提供访问策略基础能力管理入口，用于维护属性字典和策略模板。
- **FR-002**：系统必须支持属性字典管理，属性至少包含属性编码、属性名称、属性类型、可选值、启用状态和说明。
- **FR-003**：系统必须保证属性编码可用于唯一识别访问树叶子节点引用的属性，并避免 DATA_OWNER 使用未开放或已禁用属性创建新策略。
- **FR-004**：系统必须支持策略模板管理，模板至少包含名称、描述、策略表达式、访问树 JSON 和状态。
- **FR-005**：系统必须保证 PLATFORM_ADMIN 管理的是规则、属性和模板，而不是替 DATA_OWNER 创建具体租户文件访问策略。
- **FR-006**：系统必须保证 PLATFORM_ADMIN 不因管理属性或模板而获得租户文件访问权或解密权。
- **FR-007**：系统必须为 DATA_OWNER 提供“访问策略构建”和“我的访问策略”能力。
- **FR-008**：DATA_OWNER 必须只能基于平台启用并开放的属性字典构建属性叶子节点。
- **FR-009**：DATA_OWNER 必须能够选择平台启用的策略模板作为新策略起点，并在此基础上继续编辑。
- **FR-010**：访问树编辑器必须支持 AND 逻辑节点、OR 逻辑节点和属性叶子节点。
- **FR-011**：访问树编辑器必须支持新增节点、删除节点、修改节点类型、修改属性节点配置、移动节点位置和自动整理布局。
- **FR-012**：访问树画布必须支持拖动、缩放、自动居中、重置视图、节点选中高亮和节点 hover 快捷操作；在成本可接受时应提供小地图。
- **FR-013**：系统必须在访问树编辑过程中实时生成结构化访问树 JSON。
- **FR-014**：系统必须在访问树编辑过程中实时生成可读策略表达式。
- **FR-015**：系统必须提供访问树 JSON 预览、策略表达式预览和校验错误提示。
- **FR-016**：系统必须支持访问策略保存前校验，校验规则至少包括根节点非空、AND/OR 子节点数量、叶子节点属性、操作符、属性值、孤立节点、循环引用、属性启用状态和属性开放状态。
- **FR-017**：系统必须在校验失败时给出明确失败原因，并在可视化画布中标识对应问题节点。
- **FR-018**：DATA_OWNER 保存策略时，系统必须保存策略名称、策略描述、策略表达式、访问树 JSON、租户标识、创建者标识和状态。
- **FR-019**：系统必须支持 DATA_OWNER 查看、编辑、删除自己创建的访问策略。
- **FR-020**：系统必须支持根据已保存的访问树 JSON 回显可视化访问树，并恢复节点配置、表达式预览和 JSON 预览。
- **FR-021**：TENANT_ADMIN 必须只能查看本租户访问策略列表和详情，不能编辑或删除 DATA_OWNER 创建的策略。
- **FR-022**：DATA_VISITOR 不得创建、编辑、删除或进入访问策略构建页面。
- **FR-023**：系统必须基于当前用户角色动态生成菜单，并保证菜单隐藏仅作为体验优化，所有受保护操作仍必须在服务端校验权限。
- **FR-024**：系统必须对所有租户相关访问策略操作进行租户边界校验，防止跨租户访问。
- **FR-025**：系统必须对 DATA_OWNER 的访问策略写操作进行所有者边界校验，防止修改或删除他人策略。
- **FR-026**：系统必须在保存访问策略前重新校验访问树 JSON，不能完全信任前端生成结果。
- **FR-027**：系统必须校验访问树中引用的属性来自平台启用的属性字典。
- **FR-028**：系统必须校验策略表达式与访问树 JSON 的基本一致性；若严格一致性校验不在本阶段完成，系统必须以后端校验通过的访问树 JSON 为准生成标准策略表达式。
- **FR-029**：系统必须为访问树校验规则提供自动化测试覆盖。
- **FR-030**：本功能不得实现文件上传、文件下载、AES 加密、CP-ABE 加密、密钥生成、用户私钥分发或策略满足性判断。
- **FR-031**：新增或修改的 Go 业务代码必须满足项目中文注释规范：每个函数和方法都有前置注释，导出标识符符合 GoDoc 前缀规范，核心实体、Handler、Service、Repository、Middleware 注释解释业务语义、副作用和安全边界。

### 关键实体

- **策略属性**：平台开放给 DATA_OWNER 选择的属性定义。关键属性包括属性编码、属性名称、属性类型、可选值、启用状态和说明。属性编码参与访问树叶子节点引用和后续策略表达式生成。
- **策略模板**：PLATFORM_ADMIN 维护的可复用策略起点。关键属性包括模板名称、模板描述、模板表达式、模板访问树 JSON 和状态。模板只提供构建入口，不代表具体租户文件访问授权。
- **访问策略**：DATA_OWNER 在某个租户下创建的具体策略。关键属性包括策略名称、策略描述、策略表达式、访问树 JSON、租户标识、创建者标识和状态。租户标识用于租户隔离，创建者标识用于 DATA_OWNER 所有者边界。
- **访问树节点**：构成访问策略的可视化节点。节点类型包括 AND、OR 和属性叶子节点。逻辑节点通过子节点表达组合关系，属性叶子节点通过属性、操作符和值表达条件。
- **角色菜单项**：基于当前用户角色生成的功能入口。菜单项必须与服务端权限边界保持一致，但不能替代服务端鉴权。

### 宪章对齐（必填）

- **混合加密与真实 CP-ABE**：本功能只构建访问策略表达、访问树 JSON、策略表达式和策略管理能力，不执行文件内容 AES-GCM 加密、DEK 封装、RSA-OAEP 或 CP-ABE 加解密。后续接入文件加密共享时，本功能生成的策略只能作为 CP-ABE 策略输入，真实加解密仍必须由 Crypto 模块和真实 Go CP-ABE 库完成。
- **模块边界**：本功能主要落在 Policy 模块和权限/菜单相关边界。访问树解析、校验、表达式生成和策略模板能力必须归属 Policy 模块；鉴权和租户上下文由既有认证授权边界提供；不得把密码算法逻辑放入 Handler、Service 或前端访问树编辑器。
- **可解释性**：系统必须通过可视化访问树、表达式预览、JSON 预览和校验提示解释策略由哪些属性和逻辑关系组成，并能说明为什么某个策略无效或不能保存。
- **Go 注释要求**：本功能会新增或修改 Go 业务代码，因此后续 plan 必须说明注释策略，tasks 必须包含“关键注释和可读性检查”。实现时所有函数/方法必须有中文前置注释，导出标识符必须符合 GoDoc，实体字段、Handler、Service、Repository、Middleware 必须解释业务语义、副作用、权限边界和租户隔离边界。

## 成功标准（必填）

### 可衡量结果

- **SC-001**：DATA_OWNER 在已有属性字典和模板的前提下，能够在 5 分钟内完成一条包含至少 1 个逻辑节点和 2 个属性叶子节点的访问策略创建、保存和回显。
- **SC-002**：95% 的访问树校验失败场景能够定位到具体失败原因，并在界面中标识对应节点或字段。
- **SC-003**：PLATFORM_ADMIN、DATA_OWNER、TENANT_ADMIN、DATA_VISITOR 四类角色的菜单展示和受保护操作权限测试全部通过。
- **SC-004**：跨租户访问、DATA_OWNER 修改他人策略、DATA_VISITOR 写操作、PLATFORM_ADMIN 调用 DATA_OWNER 策略写操作等越权场景全部被拒绝。
- **SC-005**：访问树 JSON 回显成功率达到 100%，已保存策略重新打开后节点类型、属性、操作符、值、表达式和校验状态保持一致。
- **SC-006**：访问树校验规则具备自动化测试覆盖，核心合法与非法样例均可被稳定验证。
- **SC-007**：新增或修改的 Go 业务代码在实现阶段通过格式化与自动化测试，且完成关键注释和可读性检查。
- **SC-008**：功能验收时，不包含文件上传、文件下载、AES 加密、CP-ABE 加密、密钥生成、私钥分发和策略满足性判断，避免扩大本阶段范围。

## 假设

- 现有登录、多租户、租户成员、角色分配和基础鉴权能力可复用。
- 当前角色集合为 PLATFORM_ADMIN、TENANT_ADMIN、DATA_OWNER、DATA_VISITOR，本阶段不新增角色。
- 第一阶段访问树属性叶子节点操作符只要求支持 `=` 和 `!=`，数值比较操作符作为后续扩展预留。
- 本阶段访问树只要求支持 AND、OR 和属性叶子节点，不包含阈值门节点；阈值门可在后续 LSSS 可视化增强中扩展。
- 策略表达式主要用于预览、解释和后续 CP-ABE 策略转换准备；保存时以后端校验通过的访问树 JSON 作为权威来源。
- 本功能不改变既有登录、多租户、租户成员管理和角色分配能力，只在其基础上新增菜单、策略管理和权限校验。
