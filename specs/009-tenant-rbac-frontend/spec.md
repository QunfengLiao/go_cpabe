# 功能规格：租户 RBAC 前端授权与角色管理

**功能分支**：`009-tenant-rbac-frontend`

**创建日期**：2026-07-11

**状态**：草稿

**输入**：用户描述：“基于后端租户级 RBAC，完成前端角色管理、成员多角色分配、菜单/路由/按钮按 permission code 授权、账号与租户切换时授权上下文刷新，并满足给定验收标准。规格文档和验收场景使用中文编写。”

## 当前前端现状盘点

- 菜单配置目前集中在 `desktop/src/renderer/src/components/AppLayout.tsx`，租户管理、策略管理和平台管理菜单仍主要根据 `TENANT_ADMIN`、`DO`、`PLATFORM_ADMIN` 等固定角色 code 显示。
- 路由守卫目前集中在 `desktop/src/renderer/src/main.tsx`、`RequireTenantRole.tsx`、`RequirePlatformAdmin.tsx`、`RequireAuth.tsx`，其中租户业务路由仍通过固定角色数组判断，未形成统一 permission code 守卫。
- 当前用户、租户、角色和权限状态集中在 `AuthContext.tsx`、`authRuntime.ts`、`authStorage.ts`。系统已经有 `permissions`、按账号保存租户上下文、账号切换清空旧上下文等雏形，但仍存在由固定角色回退生成伪权限的逻辑。
- 请求租户上下文与租户请求头主要在 `request.ts` 中处理，`X-Tenant-Id` 来自当前授权运行时；必须避免账号切换、租户切换或网络失败时继续携带旧账号或旧租户上下文。
- 成员角色弹窗位于 `desktop/src/renderer/src/components/TenantMemberRoleDialog.tsx`，当前只支持在 `DO` 和 `DU` 中单选，并且展示逻辑会按固定角色 code 解释成员角色。
- 租户成员页位于 `TenantMembersPage.tsx`，当前修改角色按钮由 `TENANT_ADMIN` 判断，成员角色保存仍调用单角色业务角色接口。
- 组织管理页和策略相关页面已有操作按钮和写操作，需要从固定角色判断迁移到 permission code，同时保证现有租户切换、组织和策略页面不回归。
- 当前前端测试使用 Vitest，已有 `authStorage.test.ts` 和 `request.test.ts` 可作为授权上下文、租户请求头和缓存隔离测试基础。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 用户按真实权限进入功能（优先级：P1）

租户成员登录或切换租户后，需要看到与当前账号、当前租户、当前权限集合一致的菜单、路由和按钮；无权限用户即使手动输入 URL，也不能进入受保护页面。

**为什么优先**：这是从固定角色授权迁移到 permission code 授权的安全体验基础，直接影响角色管理、成员管理、组织和策略页面是否可被正确访问。

**独立测试**：准备拥有 `tenant.role.read` 的用户和不拥有该权限的用户，分别登录、刷新页面、手动输入角色管理页面 URL，验证菜单、路由和提示行为。

**验收场景**：

1. **给定** 当前用户拥有 `tenant.role.read`，**当** 打开应用并完成租户授权上下文加载，**那么** 角色管理菜单可见。
2. **给定** 当前用户没有 `tenant.role.read`，**当** 打开应用，**那么** 角色管理菜单不可见。
3. **给定** 当前用户没有 `tenant.role.read`，**当** 手动输入角色管理页面 URL，**那么** 系统阻止进入并显示无权限提示或跳转到安全页面。
4. **给定** 当前用户只有 `tenant.role.read`，**当** 打开角色管理页面，**那么** 用户只能查看角色，不能创建、编辑、禁用角色或修改权限。
5. **给定** 当前用户拥有 `tenant.role.manage`，**当** 打开角色管理页面，**那么** 用户可以看到创建角色和可管理自定义角色的操作入口。

---

### 用户故事 2 - 授权上下文在账号和租户生命周期中安全刷新（优先级：P1）

用户登录、切换账号、切换租户、退出登录或刷新 Electron 页面时，前端需要及时清空旧授权并重新加载当前账号和租户的真实权限，避免短暂显示旧菜单或携带旧 `tenant_id`。

**为什么优先**：多账号、多租户场景下旧授权泄漏会造成误导性菜单、错误请求头和潜在越权请求，是本功能的核心风险。

**独立测试**：准备账号 1 和账号 2，并让二者属于不同租户、拥有不同权限；执行登录、账号切换、租户切换、刷新和退出登录，验证授权状态、菜单和请求头。

**验收场景**：

1. **给定** 登录账号 1，**当** 授权上下文加载完成，**那么** 只显示账号 1 在当前租户下可访问的菜单。
2. **给定** 已登录账号 1，**当** 切换到账号 2，**那么** 切换过程中不短暂显示账号 1 的权限菜单。
3. **给定** 已切换到账号 2，**当** 发起租户业务请求，**那么** 请求不会继续携带账号 1 的 `tenant_id` 或权限缓存。
4. **给定** 当前用户切换租户，**当** 新租户授权上下文加载完成，**那么** 权限、菜单、路由和按钮立即按新租户重新计算。
5. **给定** 用户退出登录，**当** 返回登录或选择租户入口，**那么** 当前角色、权限、租户和授权状态完全清空。
6. **给定** Electron 页面刷新，**当** 应用恢复登录态和租户上下文，**那么** 正确恢复当前账号、当前租户和权限集合，不使用其他账号的缓存。
7. **给定** 授权上下文加载网络失败，**当** 页面仍处于错误或待恢复状态，**那么** 系统不得保留为授权成功状态，也不得放行受保护菜单、路由或按钮。

### 用户故事 3 - 租户管理员维护租户角色（优先级：P1）

拥有角色管理权限的租户成员需要在租户内查看内置角色和自定义角色，创建业务角色，并维护自定义角色的名称、描述和权限集合。

**为什么优先**：后端 RBAC 已提供角色和权限事实源，前端必须提供可演示、可验收的角色管理闭环，才能证明授权从固定角色迁移到可配置权限。

**独立测试**：使用拥有 `tenant.role.manage` 的用户创建一个自定义角色，修改名称、描述和权限，验证角色出现在业务角色分组；再使用只拥有 `tenant.role.read` 的用户验证只读体验。

**验收场景**：

1. **给定** 当前用户拥有 `tenant.role.manage`，**当** 提交合法角色名称、code、描述和权限集合，**那么** 自定义角色创建成功。
2. **给定** 自定义角色创建成功，**当** 返回角色列表，**那么** 该角色出现在业务角色分组。
3. **给定** 租户 A 和租户 B 都有角色列表，**当** 分别查询角色，**那么** 不同租户的角色列表相互隔离。
4. **给定** 角色是系统内置角色，**当** 用户查看列表和详情，**那么** 角色显示只读状态和系统内置标识。
5. **给定** 角色是系统内置角色，**当** 用户尝试编辑权限或禁用，**那么** 前端不提供可执行入口，并能展示其由系统管理。
6. **给定** 角色是租户自定义角色且当前用户拥有 `tenant.role.manage`，**当** 修改名称、描述和权限，**那么** 保存成功后列表和详情展示最新信息。
7. **给定** 权限选择器加载权限目录，**当** 用户为租户自定义角色配置权限，**那么** 选择器不显示平台权限。
8. **给定** 后端返回角色 code 冲突，**当** 创建角色失败，**那么** 前端显示明确的 code 冲突错误。
9. **给定** 角色名称被修改，**当** 重新判断菜单、路由和按钮权限，**那么** 权限判断不受角色名称变化影响。

### 用户故事 4 - 租户管理员为成员分配多个角色（优先级：P1）

拥有成员管理权限的租户成员需要在成员角色弹窗中为成员选择多个租户角色，包括同时选择 `DO` 和 `DU`，并保存完整角色集合。

**为什么优先**：成员多角色是权限并集计算的前提，也修复当前 `DO/DU` 单选互斥的前端限制。

**独立测试**：打开成员角色弹窗，同时选择 `DO`、`DU` 和一个自定义业务角色，保存后验证成员列表和当前授权上下文刷新。

**验收场景**：

1. **给定** 当前用户拥有 `tenant.member.manage`，**当** 打开成员角色弹窗，**那么** 弹窗支持多选角色。
2. **给定** 弹窗中存在 `DO` 和 `DU`，**当** 用户同时选择二者并保存，**那么** 保存成功且二者都保留。
3. **给定** 当前租户存在可分配角色，**当** 打开成员角色弹窗，**那么** `PLATFORM_ADMIN` 不出现在租户角色选项中。
4. **给定** 当前用户是普通成员且没有 `tenant.member.manage`，**当** 查看其他成员，**那么** 不能修改其他成员角色。
5. **给定** 操作会移除最后一名 `TENANT_ADMIN`，**当** 后端拒绝保存，**那么** 前端显示明确的最后管理员保护错误。
6. **给定** 操作者修改了自己的角色，**当** 保存成功，**那么** 当前权限立即刷新。
7. **给定** 操作者保存后失去当前页面权限，**当** 授权刷新完成，**那么** 系统安全跳转到第一个有权限页面或无权限页面。
8. **给定** 用户关闭弹窗且没有保存，**当** 回到成员列表，**那么** 弹窗中的临时选择不会污染外部成员状态。

---

### 用户故事 5 - 错误提示和既有页面不回归（优先级：P2）

用户在权限不足、网络失败、并发变更、角色冲突和后端 403 等场景下，需要看到明确提示；同时原有租户切换、组织和策略页面应继续可用并按新权限模型控制操作入口。

**为什么优先**：清晰错误提示能降低误操作和测试成本；组织、策略等既有页面是当前系统的核心演示路径，不能因 RBAC 前端改造破坏。

**独立测试**：分别模拟 403、409、网络失败、禁用角色、权限变化和租户切换，验证错误提示、授权状态和页面回归。

**验收场景**：

1. **给定** 后端返回 403，**当** 前端收到响应，**那么** 用户看到正确的权限不足提示。
2. **给定** 网络请求失败，**当** 授权上下文或角色操作未完成，**那么** 系统不会错误保留为授权成功状态。
3. **给定** 菜单、路由和按钮需要授权，**当** 判断可见性或可操作性，**那么** 不再使用固定角色 code 作为授权来源。
4. **给定** 原有租户切换、组织和策略页面，**当** 用户按各自权限访问和操作，**那么** 页面不出现权限模型改造导致的回归。

### 边界情况

- 授权上下文加载中、失败、超时或被账号切换打断。
- 当前账号已经没有任何可用租户，或当前租户被禁用、移除。
- 当前用户在一个租户有权限，在另一个租户没有权限。
- 后端返回空权限集合、空角色集合、禁用角色、已删除角色或跨租户角色。
- 角色列表或权限目录加载后被其他管理员并发修改。
- 创建角色时 code 格式错误或与当前租户已有角色冲突。
- 只读用户直接调用创建、编辑、禁用或权限保存入口。
- 内置角色详情可查看，但任何编辑权限、禁用或覆盖默认权限的操作都必须不可执行。
- 成员当前拥有后续被禁用的角色，需要清晰展示并避免再次选择。
- 用户修改自己的角色后，当前页面、菜单和请求上下文需要立即与最新权限一致。
- 平台管理员和租户成员权限作用域不同，平台权限不得出现在租户角色权限选择器中。

## 需求 *(必填)*

### 功能需求

- **FR-001**：前端必须建立统一授权上下文，至少包含当前用户、当前租户、当前租户角色、当前权限集合、授权状态、授权所属用户和授权所属租户。
- **FR-002**：授权状态必须能区分空闲、加载中、已就绪和错误状态，并在加载中或错误时阻止受保护菜单、路由和按钮使用旧权限放行。
- **FR-003**：前端必须提供统一权限判断能力，包括判断单个权限、任一权限、全部权限、刷新授权上下文和清空授权上下文。
- **FR-004**：业务组件可以展示角色信息，但不得直接遍历或匹配固定角色 code 作为功能授权来源。
- **FR-005**：登录成功并解析当前租户后，前端必须加载当前用户在当前租户的授权上下文。
- **FR-006**：账号切换开始时，前端必须立即清空旧账号角色、权限、租户上下文和受保护菜单，待新账号授权上下文就绪后再渲染受保护能力。
- **FR-007**：账号切换后，前端不得使用账号 1 的 `tenant_id`、权限集合或菜单状态请求账号 2 的接口。
- **FR-008**：租户切换开始时，前端必须清空旧租户权限并进入加载状态；新租户授权就绪后，菜单、路由和按钮必须重新计算。
- **FR-009**：退出登录后，前端必须清空当前用户、当前租户、当前角色、当前权限和授权加载状态。
- **FR-010**：Electron 页面刷新后，前端必须恢复正确账号和租户的授权上下文；无法恢复时必须进入未授权或错误状态。
- **FR-011**：菜单配置必须支持按 permission code 控制可见性；角色管理菜单至少需要 `tenant.role.read`。
- **FR-012**：父级菜单没有任何可见子菜单时必须自动隐藏。
- **FR-013**：菜单授权判断不得受侧边栏折叠、菜单展开状态或展示标签影响。
- **FR-014**：平台菜单和租户菜单必须使用不同作用域的权限判断；平台权限不得作为租户业务菜单的授权来源。
- **FR-015**：前端必须提供统一路由权限守卫，覆盖未登录、租户上下文未解析、授权加载中、授权失败、已登录但无权限和有权限场景。
- **FR-016**：无权限用户手动输入受保护页面 URL 时，前端必须显示 403 页面或跳转到安全页面，不得仅依赖菜单隐藏。
- **FR-017**：当前用户失去当前页面所需权限后，前端必须安全跳转到第一个有权限页面或无权限页面，且不得形成无限重定向。
- **FR-018**：按钮和操作入口必须按 permission code 控制可见性或可用性；角色创建、编辑、禁用和权限配置需要 `tenant.role.manage`。
- **FR-019**：成员角色修改入口必须需要 `tenant.member.manage`；普通成员不得修改其他成员角色。
- **FR-020**：组织查看和写操作必须分别受 `tenant.org.read`、`tenant.org.manage` 控制；策略查看、写入和发布必须分别受 `policy.read`、`policy.write`、`policy.publish` 控制。
- **FR-021**：前端必须新增租户角色管理页面，能区分治理角色、业务角色和能力角色，并区分系统内置与租户自定义。
- **FR-022**：角色列表至少展示角色名称、code、角色分类、内置或自定义标识、状态、权限数量、有效成员数量、描述、创建时间和操作入口。
- **FR-023**：角色列表必须支持分类筛选、状态筛选、关键字查询、详情查看、创建、编辑、配置权限和禁用操作。
- **FR-024**：创建角色表单必须包含角色名称、角色 code、描述和权限集合；不得允许用户指定作用域、角色分类、租户 ID 或内置标记。
- **FR-025**：创建成功的自定义角色必须出现在业务角色分组。
- **FR-026**：不同租户的角色列表必须相互隔离，前端不得把其他租户角色缓存并展示到当前租户。
- **FR-027**：内置角色必须显示只读状态和系统内置标识，不允许编辑权限或禁用。
- **FR-028**：自定义角色必须允许编辑名称、描述和权限；角色 code 创建后不得修改。
- **FR-029**：权限选择器必须从后端权限目录加载可选权限，并且不显示平台权限。
- **FR-030**：权限选择器必须支持按资源模块分组、分组全选、分组取消、搜索、已选择数量和未保存修改提示。
- **FR-031**：保存角色权限成功后，前端必须刷新角色详情；如果被修改角色影响当前用户权限，必须刷新当前授权上下文。
- **FR-032**：成员角色弹窗必须从单选改为多选，按治理角色、业务角色和能力角色分组展示。
- **FR-033**：成员角色弹窗中的每个角色至少展示名称、code、描述、系统内置或租户自定义标识和状态。
- **FR-034**：成员角色弹窗必须允许 `DO` 和 `DU` 同时选择并保存，不得在前端实现二者互斥。
- **FR-035**：`PLATFORM_ADMIN` 不得出现在租户成员角色选项中。
- **FR-036**：禁用角色不得被新选择；成员已经拥有但后来禁用的角色必须有清晰展示和处理方式。
- **FR-037**：成员角色保存前必须展示新增与移除角色差异，保存时提交完整角色集合。
- **FR-038**：成员角色更新成功后，必须刷新成员列表、角色数量；如果修改的是当前用户，还必须刷新当前授权上下文。
- **FR-039**：移除最后一名 `TENANT_ADMIN`、角色 code 冲突、无权限、角色不存在、角色禁用、权限变更、网络失败等错误必须展示明确提示，不得统一显示为“操作失败”。
- **FR-040**：后端返回 403 时，前端必须展示权限不足提示，并保持授权状态与后端拒绝结果一致。
- **FR-041**：网络失败或授权上下文加载失败时，前端不得把旧权限保留为授权成功状态。
- **FR-042**：菜单、路由和按钮不得再使用 `TENANT_ADMIN`、`DO`、`DU`、`PLATFORM_ADMIN` 等固定角色 code 作为功能授权来源。
- **FR-043**：角色名称、描述或展示分组变化不得影响 permission code 判断。
- **FR-044**：本功能必须通过自动化测试或手工验收覆盖用户给定验收标准。
- **FR-045**：本阶段不得实现动态菜单数据库、角色继承、DENY、数据范围配置、平台公共模板完整管理、角色审批流或将自定义角色同步为 CP-ABE 属性。

### 关键实体

- **授权上下文**：当前账号在当前租户下的授权结果，包含权限集合、角色展示信息、所属用户、所属租户和加载状态。
- **权限**：可被菜单、路由、按钮和操作入口判断的稳定 code，例如 `tenant.role.read`、`tenant.role.manage`、`tenant.member.manage`。
- **角色**：租户内可展示和可分配的身份集合，包含系统内置角色和租户自定义角色，角色名称仅用于展示。
- **角色分组**：用于前端认知和筛选的角色分类，包括治理角色、业务角色和能力角色。
- **成员角色分配**：管理员对一个租户成员提交的完整角色集合，结果影响该成员后续权限并集。
- **受保护导航项**：需要权限才能显示或进入的菜单项、路由和按钮操作。

### 宪章对齐（必填）

- **混合加密与真实 CP-ABE**：本功能不修改文件内容 AES-GCM、DEK 封装、RSA-OAEP 或 CP-ABE 加解密逻辑。角色和权限只控制前端功能入口与后端请求体验，不代表用户一定满足 CP-ABE 文件访问策略。
- **模块边界**：本功能主要属于前端授权体验、租户角色管理、成员角色管理、组织和策略页面权限控制。密码算法逻辑不得进入前端角色管理或授权判断；后端仍是最终安全边界。
- **可解释性**：界面必须清楚解释当前用户为何能看到或不能看到角色管理、成员管理、组织和策略操作；角色列表必须区分系统内置、租户自定义、只读、禁用和权限数量。
- **Go 注释要求**：本规格主要面向前端改造。如计划阶段发现必须做最小后端适配，所有新增或修改 Go 业务代码必须满足宪章注释要求；对应 `plan.md` 必须说明注释策略，`tasks.md` 必须包含“关键注释和可读性检查”任务。

## 假设

- 后端租户级 RBAC 接口和权限目录以 `specs/008-tenant-rbac-backend` 为上游依据，前端优先复用当前租户上下文接口和当前租户角色接口。
- 前端授权控制是用户体验层，后端仍负责最终权限判断、租户隔离和最后管理员保护。
- 本阶段继续使用前端静态菜单配置，不新增 `ui_menus`、`permission_menus` 或动态菜单数据库。
- 平台权限和租户权限作用域已经能由后端权限目录区分；前端只展示当前租户可选权限。
- 角色 code 可用于展示、分组和提交角色集合，但不能作为菜单、路由或按钮的功能授权来源。
- 如果后端某个错误 code 尚未稳定，前端至少应显示后端返回的明确中文消息，并在计划阶段补充错误映射策略。

## 明确排除范围

- 后端 RBAC 表结构重设计和大规模迁移。
- 动态菜单数据库、菜单权限绑定表和 API 资源绑定表。
- 角色继承、DENY、条件权限、数据范围配置和角色审批工作流。
- 将自定义角色自动同步为 CP-ABE 属性。
- 文件、密钥、审计落库和完整平台公共模板管理页面。
- CP-ABE、RSA-OAEP、AES-GCM 和访问树/LSSS 加密实现改造。
