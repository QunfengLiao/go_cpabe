# 功能规格：基础工程初始化

**功能分支**：`main`
**创建日期**：2026-07-05
**状态**：草稿
**输入**：用户描述：“实现基础工程初始化：搭建可本地开发的全栈桌面端基础工程，包含后端服务、桌面端首页、远程 MySQL 和 Redis 配置、健康检查、API 调用封装、README、.env.example 与 .gitignore。”

## 用户场景与测试

### 主要用户故事

作为开发者，我希望拿到仓库后能够根据示例环境变量配置远程数据服务，分别启动后端服务和桌面端应用，并通过健康检查与基础首页确认本地开发环境已经可用。完成后，我还希望通过项目说明文档了解启动、测试和常见问题排查方式，为后续文件加密共享主链路开发打好基础。

### 验收场景

1. **给定**开发者已经克隆仓库并复制示例环境变量文件，**当**开发者填写远程 MySQL 连接信息并启动后端服务，**则**系统能够读取配置并在健康检查结果中展示 MySQL 连接状态。
2. **给定**开发者已经填写远程 Redis 连接信息，**当**开发者启动后端服务并访问健康检查，**则**系统能够展示 Redis 连接状态。
3. **给定**后端服务已经启动，**当**开发者访问健康检查入口，**则**开发者能够看到服务自身、MySQL、Redis 的可观察状态与失败原因摘要。
4. **给定**桌面端依赖已经安装，**当**开发者启动桌面端应用，**则**开发者能够看到一个基础首页，用于确认桌面端运行正常。
5. **给定**开发者准备继续开发业务功能，**当**开发者查看桌面端代码，**则**能够找到预留的后端调用封装入口，并理解后续业务页面如何复用它。
6. **给定**新开发者首次接触项目，**当**阅读根目录项目说明，**则**能够按文档完成环境变量配置、服务启动、测试执行和常见问题排查。

### 边界情况

- 远程 MySQL 配置缺失、格式错误、网络不可达或认证失败时，健康检查必须明确展示 MySQL 不可用，并且不得泄露真实密码或完整密钥。
- 远程 Redis 配置缺失、格式错误、网络不可达或认证失败时，健康检查必须明确展示 Redis 不可用，并且不得泄露真实密码或完整密钥。
- 未创建本地环境变量文件时，开发者必须能够通过示例文件和说明文档知道需要配置哪些项目。
- 桌面端无法连接后端时，基础首页仍应能启动，并在后续调用封装中保留清晰的错误处理边界。
- 仓库中不得提交真实服务器地址、真实账号、真实密码、访问令牌或其他敏感信息。

## 需求

### 功能需求

- **FR-001**：系统 MUST 提供一套后端基础工程入口，使开发者能够在本地启动服务并获得明确的启动结果。
- **FR-002**：系统 MUST 通过本地环境变量读取远程 MySQL 连接配置，并支持开发者使用示例环境变量文件完成配置。
- **FR-003**：系统 MUST 通过本地环境变量读取远程 Redis 连接配置，并支持开发者使用示例环境变量文件完成配置。
- **FR-004**：系统 MUST 提供健康检查能力，返回服务自身、MySQL、Redis 的可观察状态。
- **FR-005**：健康检查结果 MUST 在依赖不可用时提供可排查的失败摘要，同时 MUST 避免输出真实密码、令牌或完整连接密钥。
- **FR-006**：系统 MUST 提供桌面端基础工程入口，使开发者能够在本地启动桌面端并看到基础首页。
- **FR-007**：桌面端 MUST 预留统一的后端调用封装入口，便于后续用户、文件、策略、加密和审计功能复用。
- **FR-008**：项目 MUST 提供根目录说明文档，覆盖环境变量配置、后端启动、桌面端启动、测试运行和常见问题排查。
- **FR-009**：项目 MUST 提供示例环境变量文件，只包含占位值、默认本地开发值或说明性示例，不得包含真实服务器凭据。
- **FR-010**：项目 MUST 提供忽略规则，避免本地环境变量、依赖产物、构建产物、日志和临时文件被提交到仓库。
- **FR-011**：基础工程 MUST 明确保留后续核心模块的边界，至少覆盖用户、文件、策略、密码学、基准测试和审计能力的扩展位置。
- **FR-012**：本功能 MUST 不实现用户注册、登录、权限系统、具体业务表、默认本地数据服务容器、集群部署、持续集成、持续交付或消息队列。
- **FR-013**：项目文档 MUST 使用简体中文描述启动和排查流程，代码标识符、路径、命令、配置键、算法名称和第三方库名称可保留英文。

### 宪章对齐

- **混合加密**：本功能不实现文件内容加密和 DEK 封装，但基础工程必须为后续“文件内容使用 AES-GCM、DEK 使用 RSA/CP-ABE 封装”的主链路预留清晰模块边界。
- **真实 CP-ABE**：本功能不实现 CP-ABE 加解密，不得加入模拟 CP-ABE 结果或教学伪实现；后续 CP-ABE 能力必须接入真实 Go 密码学库。
- **可插拔算法**：本功能应在工程结构和文档中为统一 `CryptoEngine`、`RSAEngine`、`TKN20Engine` 扩展留出位置，业务入口不得提前绑定具体算法逻辑。
- **RSA 基线**：本功能不产生 RSA 与 CP-ABE 性能结论；README 可说明本阶段仅初始化工程，不进行算法对比。
- **策略可视化**：本功能不实现访问树或 LSSS 可视化，但工程边界应允许后续 Policy 模块承载策略解析、校验和可视化职责。
- **可解释性**：本功能的可解释性集中在开发环境状态：健康检查必须能说明服务、MySQL、Redis 是否可用，以及不可用时的排查方向。
- **审计与基准**：本功能不记录业务审计日志或算法基准数据，但工程结构应为后续 Benchmark 与 Audit 模块扩展留出位置。
- **语言规范**：本规格及派生文档 MUST 使用简体中文；代码标识符、路径、命令、API 路径、JSON 字段、数据库表名、Go/TypeScript 类型名、第三方库和算法名称除外。

### 关键实体

- **本地环境配置**：开发者本机用于启动服务的配置集合，包含服务端口、远程 MySQL 连接信息、远程 Redis 连接信息和桌面端访问后端所需的基础地址；真实敏感值只存在于本地环境，不进入仓库。
- **健康检查结果**：开发者用于判断本地工程是否可运行的状态集合，包含服务自身状态、MySQL 状态、Redis 状态、检查时间和安全的失败摘要。
- **桌面端基础首页**：开发者启动桌面端后看到的默认界面，用于确认桌面端工程运行正常，并为后续业务入口预留展示位置。
- **后端调用封装**：桌面端内部访问后端能力的统一入口，承载基础地址、请求发起、错误处理和后续接口扩展的约定。
- **项目启动说明**：根目录面向新开发者的说明文档，覆盖配置、启动、测试和排障流程。

## 成功标准

### 可衡量结果

- **SC-001**：新开发者在已有远程数据服务凭据的前提下，能够在 20 分钟内根据项目说明完成环境变量配置并启动后端服务。
- **SC-002**：开发者访问健康检查时，能够在一次响应中看到服务自身、MySQL、Redis 三类状态；依赖异常时，响应必须包含不泄露敏感信息的失败摘要。
- **SC-003**：开发者能够在 10 分钟内启动桌面端应用，并看到用于确认桌面端可运行的基础首页。
- **SC-004**：仓库扫描和人工检查均确认未提交真实密码、真实密钥、真实令牌或个人服务器敏感连接信息。
- **SC-005**：至少 90% 的首次启动失败可通过 README 的常见问题排查章节定位到配置缺失、网络连接、认证失败或依赖未安装等原因。
- **SC-006**：后续功能开发者能够在不重建基础工程的情况下，识别并复用后端服务入口、桌面端入口、环境配置、后端调用封装和核心模块扩展位置。

## 假设

- 开发者已经拥有可访问的远程 MySQL 和 Redis 服务，以及对应的连接凭据。
- 本阶段只验证远程依赖连接和本地开发启动，不负责创建、迁移或填充业务数据表。
- 本阶段的健康检查用于开发环境诊断，不作为生产级监控或安全审计能力。
- 示例配置只提供占位值和说明性默认值；真实凭据由开发者在本地自行维护。
- 桌面端基础首页以确认工程可运行和后续扩展为目标，不承担具体业务流程。
