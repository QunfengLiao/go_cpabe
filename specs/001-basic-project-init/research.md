# 第 0 阶段研究：基础工程初始化

## 决策 1：后端采用最小分层启动骨架

**决策**：后端使用 `cmd/server/main.go` 作为唯一启动入口，业务初始化下沉到 `internal/config`、`internal/router`、`internal/handler`、`internal/service`、`internal/repository`、`internal/model`、`internal/middleware`、`internal/pkg`。

**理由**：该结构符合 Go 项目常见分层方式，也满足宪章对模块边界的要求。`main.go` 只负责组装和启动，能避免把配置、路由、连接初始化和健康检查逻辑堆在入口文件中。

**备选方案**：

- 单文件后端：启动更快，但很快会污染入口文件，不利于后续 User、File、Policy、Crypto、Benchmark、Audit 扩展。
- 过早引入复杂分层或依赖注入框架：当前只是基础工程初始化，复杂度超出范围。

## 决策 2：配置通过 `.env` 与环境变量读取

**决策**：提供根目录 `.env.example`，后端启动时从环境变量读取配置，并允许本地开发通过 `.env` 注入；真实 `.env` 必须被 `.gitignore` 忽略。

**理由**：用户明确要求通过 `.env` 连接个人服务器上的 MySQL 和 Redis。示例文件能指导新开发者配置，同时避免提交真实凭据。

**备选方案**：

- 配置文件提交到仓库：容易泄露服务器地址和密码，不符合安全边界。
- 命令行参数：可用但字段多、启动体验差，不利于桌面端和后端统一说明。

## 决策 3：MySQL 使用 Gorm 初始化连接

**决策**：后端通过 Gorm 初始化 MySQL 8 连接，本阶段只验证连接和健康检查，不创建业务表和迁移。

**理由**：技术栈固定为 Gorm + MySQL 8，且本功能范围明确“不实现具体业务表”。保持连接初始化即可满足验收标准，同时避免过早设计用户、文件、策略等业务模型。

**备选方案**：

- 使用标准库 `database/sql`：更轻量，但偏离固定 ORM 要求。
- 创建初始化业务表：超出本功能范围，会提前进入用户、文件或权限业务设计。

## 决策 4：Redis 使用 go-redis 初始化客户端

**决策**：后端使用 go-redis 客户端连接 Redis 7，并通过健康检查执行轻量连通性验证。

**理由**：go-redis 是 Go 生态常用 Redis 客户端，适合当前连接验证和后续缓存、会话或限流扩展。

**备选方案**：

- 暂不接入 Redis：无法满足验收标准。
- 自行封装底层协议：没有必要，且增加维护风险。

## 决策 5：健康检查契约以状态诊断为核心

**决策**：`GET /health` 返回 `app`、`mysql`、`redis` 的状态、检查时间和安全失败摘要；依赖失败时不泄露密码、令牌、完整 DSN 或完整服务器敏感信息。

**理由**：健康检查是本功能最主要的可观察验收入口。结构化状态便于桌面端后续调用，也便于开发者排查远程连接问题。

**备选方案**：

- 只返回 `ok`：无法说明 MySQL 或 Redis 哪个依赖异常。
- 返回原始错误：便于调试但可能泄露敏感连接信息。

## 决策 6：桌面端采用 Electron + TypeScript + Vite

**决策**：桌面端使用 Electron + TypeScript，构建工具优先采用 Vite；不引入复杂 UI 框架。

**理由**：用户固定桌面端技术栈并要求保持初始化工程简单。Vite 能提供清晰的开发体验，基础首页和 API 封装无需复杂 UI 依赖。

**备选方案**：

- 传统 Electron 手写构建：可控但开发体验较弱。
- 引入复杂 UI 框架：当前只有基础首页和连接状态占位，收益不足。

## 决策 7：Docker Compose 不作为默认工程产物

**决策**：本功能不生成默认 MySQL 和 Redis 容器；README 可说明远程服务是默认路径，后续如需本地备用方案再单独规划。

**理由**：用户明确说明 MySQL 和 Redis 已部署在个人服务器上，Docker Compose 非默认要求。默认生成容器会混淆真实验收路径。

**备选方案**：

- 默认提供 MySQL/Redis Compose：与需求冲突，也可能让开发者误以为本功能依赖本地容器。
- 完全不提备用方案：可以，但 README 中保留一句“本阶段不默认提供容器”更利于排障。
