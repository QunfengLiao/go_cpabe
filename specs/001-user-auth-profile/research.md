# 第 0 阶段研究：用户认证与资料基础模块

## 决策：Access Token 使用 JWT，Refresh Token 使用随机字符串

**理由**：Access Token 需要在受保护接口中快速解析用户 ID、角色、凭证类型和过期时间，JWT 适合短期无状态校验。Refresh Token 需要服务端可控和可撤销，随机字符串配合 Redis 登录态可以在退出和轮换时立即失效。

**备选方案**：

- Refresh Token 也使用 JWT：便于携带 `jti` 和过期时间，但若只依赖 JWT 会削弱服务端撤销能力；仍需 Redis 登录态，因此随机字符串更简单。
- 单 Token 登录态：实现更少，但无法满足短期访问凭证和长期刷新凭证分离要求。

## 决策：Access Token 默认 15 分钟，Refresh Token 默认 7 天

**理由**：15 分钟能限制 Access Token 泄露后的可用窗口，7 天能满足演示系统的登录连续性。二者必须通过配置项调整，便于后续按环境改为 30 分钟或 14 天。

**备选方案**：

- Access Token 30 分钟：用户体验更宽松，但泄露窗口更长。
- Refresh Token 14 天：更适合频繁使用场景，但本阶段演示系统选择更保守的默认值。

## 决策：Refresh Token 默认采用轮换机制

**理由**：刷新成功后删除旧登录态并创建新登录态，可以降低 Refresh Token 被重放的风险，也满足规格中对旧 Token 失效的安全预期。

**备选方案**：

- 非轮换机制：实现更简单，刷新时只签发新的 Access Token；后续可通过 Token Store 接口扩展为轮换。若实施阶段时间不足，可作为临时降级方案，但计划目标仍为轮换。

## 决策：Redis Key 使用 `auth:refresh:{token_id}`

**理由**：`token_id` 能直接定位单个 Refresh Token 登录态，适合刷新、退出和轮换删除。Value 中保留 `session_id`，后续可以扩展同一用户多端登录、单端退出和全部退出。

**备选方案**：

- `auth:refresh:{user_id}:{session_id}`：更便于列出用户会话，但刷新时需要从 Token 中解析或查找会话标识。本阶段优先单 Token 定位，后续可增加用户会话索引。

## 决策：Redis 只保存 Refresh Token Hash

**理由**：即使 Redis 数据泄露，也不应直接暴露可用的 Refresh Token。Hash 比对可以满足刷新和退出校验。

**备选方案**：

- 明文保存 Refresh Token：实现简单，但不符合安全要求。
- 保存加密后的 Refresh Token：需要额外密钥管理，本阶段不引入。

## 决策：用户表采用 Gorm Model，并提供 SQL migration

**理由**：Gorm Model 便于业务代码读写和开发环境快速初始化；SQL migration 提供可审查、可重复执行的数据库变更记录，更适合后续团队协作和部署。

**备选方案**：

- 只使用 AutoMigrate：开发快，但数据库约束、索引和变更历史不够可控。
- 只写 SQL 不维护 Gorm 标签：会增加模型与表结构漂移风险。

## 决策：角色和状态常量放在 `internal/domain`

**理由**：角色与状态属于用户领域概念，应由领域层统一定义和校验，Handler、Service、Repository 只引用该包，避免字符串散落。

**备选方案**：

- 放在 Handler 或 Service：会导致跨层复用困难。
- 放在数据库模型包：容易把持久化细节和领域规则绑定过紧。

## 决策：对外使用 `UserDTO` 映射用户资料

**理由**：`UserDTO` 明确列出可返回字段，从结构层面避免 `password_hash` 和 `avatar_object_key` 被序列化给前端。

**备选方案**：

- 直接返回 Gorm Model 并用 JSON tag 隐藏字段：依赖标签正确性，后续新增字段时更容易误泄露。

## 决策：头像存储使用 Storage 接口和 `LocalStorage`

**理由**：本阶段只需要保存到 `uploads/avatars`，但项目后续可能替换为 MinIO、腾讯云 COS、阿里云 OSS。Storage 接口可以让 Service 不关心底层存储类型。

**备选方案**：

- Handler 直接保存文件：实现快，但违反分层要求。
- 立即接入对象存储：超出本阶段范围，增加部署复杂度。

## 决策：头像文件名使用用户 ID、时间戳和随机字符串

**理由**：该命名方式能避免并发上传冲突，也便于排查某个用户的头像文件。保留后缀可以支持静态访问和内容类型识别。

**备选方案**：

- 使用原始文件名：存在冲突和路径污染风险。
- 只使用随机 UUID：足够安全，但不便于人工排查；仍可作为随机串来源。

## 决策：统一错误响应不暴露内部细节

**理由**：认证模块涉及账号枚举、Token 校验、Redis 和文件系统错误。前端需要稳定错误码和可展示消息，内部细节应进入日志和审计，不直接返回。

**备选方案**：

- 直接返回底层错误：调试方便，但会泄露系统内部状态。
