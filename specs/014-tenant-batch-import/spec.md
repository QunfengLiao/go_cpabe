# 功能规格：租户数据批量导入

**功能分支**：`014-tenant-batch-import`

**创建日期**：2026-07-17

**状态**：已就绪

**输入**：用户需求：在现有 CP-ABE 多租户系统中实现租户用户和组织架构的 Excel 批量导入、模板下载、预校验、导入结果、错误报告和审计。

## 用户场景与测试

### 用户故事 1：导入组织架构（优先级：P1）

租户管理员下载组织模板，填写组织编码、名称和父子关系后上传，先查看逐行校验结果，再确认以一个原子批次创建或更新组织。

**优先级原因**：组织是用户归属和属性同步的基础，先完成组织导入可以直接支撑租户初始化和后续用户导入。

**独立测试**：使用包含父组织晚于子组织、重复编码和循环关系的模板，验证预览能定位错误，合法批次确认后组织树层级正确，异常时无部分写入。

**验收场景**：

1. **给定**当前租户管理员已登录，**当**下载并上传合法组织模板，**那么**系统返回批次和逐行“待新增/待更新”预览，数据库不发生业务数据写入。
2. **给定**子组织行出现在父组织行之前，**当**管理员确认导入，**那么**系统按依赖关系写入并生成正确的组织树与路径。
3. **给定**文件包含重复编码、跨租户父组织或循环父子关系，**当**执行预校验，**那么**错误包含 Excel 行号、字段和可操作原因，确认按钮不可用。
4. **给定**正式写入任意一行失败，**当**批次完成，**那么**组织创建、更新及关联属性值全部回滚，批次标记为失败并记录审计。

### 用户故事 2：导入租户用户（优先级：P1）

租户管理员下载用户模板，批量填写用户资料、组织、租户角色和属性后上传，查看新增/更新/跳过/错误结果，再确认导入。

**优先级原因**：用户导入是租户数据初始化的主要工作量，必须复用现有租户成员、角色和属性同步规则，不能形成第二套权限逻辑。

**独立测试**：使用新用户、其他租户已有系统用户、当前租户已有成员、非法角色和非法属性，验证预览分类、确认后的成员关系/角色/属性一致性及租户隔离。

**验收场景**：

1. **给定**用户名不存在且组织、角色和属性合法，**当**确认导入，**那么**系统创建账号、加入当前租户并同步组织、角色和属性，密码不出现在任何响应、日志或审计详情。
2. **给定**用户名已存在但不属于当前租户，**当**确认导入，**那么**系统只将该系统用户加入当前租户，不修改其他租户关系。
3. **给定**用户已属于当前租户，**当**确认导入，**那么**系统更新当前租户成员资料并通过现有角色服务替换多个租户角色。
4. **给定**文件引用平台角色、其他租户角色或不允许的属性，**当**预校验，**那么**对应行失败且不能确认。
5. **给定**租户管理员下载用户模板，**当**使用 Excel 打开，**那么**首个可见区域展示标题、填写说明和中文表头，用户可通过筛选、冻结窗格以及角色/成员状态下拉完成录入，同时隐藏的第一行仍保存原有英文字段 key。

### 用户故事 3：导入批次与错误报告（优先级：P2）

租户管理员可以查看本人当前租户的历史导入批次、批次详情及错误报告，页面刷新后可凭批次编号恢复状态。

**优先级原因**：批量操作需要可追踪、可恢复和可解释，错误报告能降低重复修改和人工排查成本。

**独立测试**：创建成功和失败批次，分别验证列表、详情、错误报告下载、当前租户隔离和操作者隔离。

**验收场景**：

1. **给定**批次已预校验或完成，**当**管理员查询批次详情，**那么**系统返回统计、状态和逐行处理结果。
2. **给定**存在失败行，**当**下载错误报告，**那么**报告至少包含原始行号、关键字段、错误字段和原因，并对公式前缀进行安全转义。
3. **给定**批次属于其他用户、其他租户或已过期，**当**尝试确认或查看，**那么**系统拒绝请求且不泄露批次内容。

### 边界情况

- 只接受 `.xlsx`；文件大小和数据行数由后端配置限制，超限在解析前拒绝。
- 标题行缺失、必要字段被修改、空文件、损坏 Excel 和公式注入必须返回可定位错误。
- 新版企业模板与旧版“第一行字段 key、第二行数据”模板必须同时可解析；装饰行只影响展示，不得改变字段映射或数据行的真实 Excel 行号。
- 组织父节点可引用数据库已有节点或同批次节点，导入排序不能依赖 Excel 行顺序。
- 组织停用时若仍有成员或子组织，批次不得把它变成不合法状态。
- 同一批次重复确认必须幂等拒绝或返回已完成结果，不得重复授予角色或创建成员关系。
- 预校验后源文件哈希或当前数据版本变化时，确认必须失败并要求重新校验。

## 需求

### 功能需求

- **FR-001**：系统必须仅允许当前租户管理员下载用户和组织架构 `.xlsx` 模板；用户模板必须包含“用户导入”“字段说明”“数据字典”三个工作表、标题、填写说明、中文展示表头和虚构示例数据，且不得包含真实生产数据。
- **FR-001A**：用户模板第一行必须保留既有英文字段 key 并默认隐藏；首个可见标题区必须显示“用户批量导入模板”和必填提示，中文表头必须使用星号标记用户名、显示名称、组织编码和角色编码。既有 `role_codes`、`employee_no`、`attributes` 等系统 key 不得改名。
- **FR-001B**：用户模板必须设置适合阅读的字体、主题色、列宽、行高、边框、自动换行、自动筛选和冻结窗格；`role_codes` 与 `member_status` 数据列必须分别提供角色组合和成员状态下拉校验。
- **FR-001C**：字段说明工作表必须说明系统字段、是否必填、示例和填写规则；数据字典工作表必须列出 `DO`、`DU`、`TENANT_ADMIN`、`ACTIVE`、`DISABLED` 的中文含义。
- **FR-002**：系统必须从可信租户上下文取得 `tenant_id`，不接受客户端传入租户 ID 作为业务边界；所有导入查询和写入必须带当前租户条件。
- **FR-003**：系统必须先解析和预校验上传文件，再由管理员确认后写入业务数据库；上传和预校验阶段不得修改用户、成员、角色、组织或属性事实。
- **FR-004**：系统必须校验扩展名、文件大小、行数、标题、必填字段、枚举、格式、重复值、组织引用、角色范围、属性范围和跨租户引用，并返回原始行号、字段和明确原因。
- **FR-005**：组织导入必须检测重复 `org_code`、自引用、循环、非法父组织和非法负责人，按父子依赖排序，并在一个事务中新增或更新允许修改的字段；导入不得删除组织。
- **FR-006**：用户导入必须区分系统用户和当前租户成员，支持新建用户、加入当前租户和更新当前租户成员；角色必须调用现有租户角色分配能力，一个用户可拥有多个租户角色。
- **FR-007**：用户导入必须同步现有组织成员关系和用户属性规则，不得授予 `PLATFORM_ADMIN`，不得写入其他租户的组织、角色或属性。
- **FR-008**：每个导入批次必须绑定当前租户、当前用户、文件哈希和短期有效批次编号，并在确认时重新校验归属、过期、篡改、当前数据状态和重复确认。
- **FR-009**：正式导入必须采用批次原子事务；任意一条业务写入失败时整体回滚，代码结构保留部分成功扩展点但默认关闭。
- **FR-010**：系统必须记录上传校验、确认、成功、失败、模板下载和错误报告下载审计；密码、原始敏感属性、临时路径和原始错误不得进入日志、错误报告或审计详情。
- **FR-011**：系统必须提供模板下载、用户/组织预校验、确认、批次列表、批次详情和错误报告下载接口，并由租户管理员权限保护。
- **FR-012**：上传步骤必须保证已选文件、校验状态、错误反馈和操作按钮互不遮挡；主操作必须明确表达“校验成功后进入数据预览”，失败原因必须保留在抽屉内而不是仅显示短暂消息。
- **FR-013**：默认配置必须支持单批次 10000 行用户数据；大批量初始密码摘要计算必须限制并发，既避免逐行串行阻塞，也不能无界占用服务端 CPU。
- **FR-012**：前端必须提供四步导入体验，展示统计卡片、逐行预览、错误筛选、动作筛选、防重复提交、错误报告下载、导入结果和完成后的用户/组织列表刷新。
- **FR-013**：临时上传文件必须使用随机文件名，解析后及时清理；错误报告中以 `=、+、-、@` 开头的内容必须安全转义。
- **FR-014**：实现必须保留现有单个用户、组织、角色和属性功能，并补充组织顺序、循环、跨租户、权限、批次和事务回滚测试。

### 关键实体

- **导入批次**：记录批次编号、租户、操作者、类型、文件摘要、统计、状态、时间和失败原因。
- **导入行结果**：保存原始行号、关键字段、规范化数据、动作、校验状态和脱敏错误列表，作为确认阶段的可信输入。
- **组织导入行**：描述组织编码、名称、父编码、类型、排序、负责人、状态和说明。
- **用户导入行**：描述用户名、展示资料、组织编码、租户角色、成员状态、职位、工号、属性和初始密码；初始密码只用于创建时，不持久化明文。

### 宪章对齐

- **混合加密与真实 CP-ABE**：本功能不触及文件内容加密或 DEK 封装，因此不改变 AES-GCM、RSA-OAEP 和真实 CP-ABE 边界。
- **模块边界**：导入解析与业务规则放在 Import 服务，用户/组织/角色/属性仍通过现有模块能力协作，Handler 只负责请求、响应和文件流。
- **可解释性**：逐行动作、字段错误、导入统计、批次状态和审计让管理员理解数据为何新增、更新、跳过或失败。
- **Go 注释要求**：新增 Go 领域实体、Repository、Service、Handler 和迁移相关函数均需有中文前置注释；导出标识符遵守 GoDoc，并说明租户、安全和事务边界。

## 成功标准

### 可衡量结果

- **SC-001**：租户管理员可在一个导入流程内下载模板、上传、预览并确认合法的用户或组织批次，且上传后业务数据不提前变化。
- **SC-002**：100% 的校验错误都能定位到 Excel 原始行号和字段，错误报告与页面展示的错误数量一致。
- **SC-003**：合法组织批次无论父子行顺序如何，确认后组织树层级、路径和组织属性值均正确。
- **SC-004**：任何正式导入异常都不会留下该批次的部分用户、成员、组织、角色或属性写入。
- **SC-005**：跨租户、平台角色、越权批次和过期批次请求全部被拒绝，且响应不泄露其他租户数据。
- **SC-006**：导入成功后用户列表、组织树、成员数量和角色展示在一次刷新中与后端事实一致。
- **SC-007**：用户模板在 Microsoft Excel 或兼容软件中打开时，三张工作表结构完整，标题和表头无截断，筛选、冻结窗格以及角色/成员状态下拉可用，且生成模板可被后端解析器无损回读。
- **SC-008**：上传页在有文件、校验中和校验失败三种状态下均无内容遮挡；校验成功自动进入预览，失败时页面持续显示可操作原因。
- **SC-009**：默认配置可接收恰好 10000 行的数据文件，超过限制时返回稳定的行数超限错误码。

## 假设

- 现有认证、租户上下文、租户角色和组织属性服务继续作为唯一业务规则来源。
- 默认批次有效期为 30 分钟；默认限制由环境变量配置，前端只展示后端返回的限制。
- 初始实现使用内存保存批次解析行，并受最大行数配置保护；后续可替换为对象存储或临时表以支持更大文件。
- 用户导入要求引用的组织在当前租户中已存在；本功能不把用户和组织导入合并为跨类型批次。
- 本版本默认批次原子导入，不开放部分成功选项。

## 增量用户故事 4：可观察、可恢复的万级异步导入（优先级：P1）

租户管理员确认一个已通过预校验的万级批次后，应立即看到任务已被系统受理，并能持续查看处理阶段、完成数量和最终结果；关闭抽屉或短暂离开页面后重新进入，仍可恢复同一批次的进度，而不是依赖一次长时间保持的请求。

**优先级原因**：同步长请求和逐行写入会让管理员误以为点击无效，也会放大超时、重复提交、服务重启和数据库长事务风险，是万级批量导入可用性的关键阻塞点。

**独立测试**：上传并确认 10000 行合法用户批次，确认操作在 1 秒内显示已受理；导入期间关闭并重新打开抽屉，页面恢复批次状态；重复确认不产生第二次导入；任务成功后用户、租户成员、角色和组织关系完整一致。

**验收场景**：

1. **给定**批次已通过预校验，**当**管理员点击确认导入，**那么**系统立即返回已受理状态，并在后台继续处理。
2. **给定**批次正在后台处理，**当**管理员查看批次，**那么**页面显示当前阶段、已处理数量、总数量和百分比，并在完成后自动展示结果。
3. **给定**同一批次已排队、处理中或已完成，**当**管理员重复确认，**那么**系统返回当前批次状态，不创建重复任务也不重复写入。
4. **给定**后台处理进程在任务期间重启，**当**任务租约失效，**那么**系统能够重新领取任务并依据幂等边界安全恢复或明确失败，不留下不可解释的中间状态。
5. **给定**正式业务写入任一步骤失败，**当**批次结束，**那么**本批次对用户、成员、角色、组织和属性的业务修改全部回滚，批次保留脱敏失败状态。

### 增量功能需求

- **FR-015**：确认导入必须转换为后台批次执行；确认请求不得承担万级数据的完整业务写入，受理成功后必须立即返回可查询的批次状态。
- **FR-016**：批次必须支持排队、处理中、成功、失败和过期等明确状态，并记录当前阶段、已处理数量、总数量及最近心跳时间。
- **FR-017**：同一批次的重复确认必须幂等；系统必须防止多实例或重复点击同时执行同一批次。
- **FR-018**：后台任务必须具备持久化领取、租约和失效恢复能力；服务进程重启不得导致任务永久停留在处理中。
- **FR-019**：正式业务数据仍必须整批提交；后台准备过程可以分段执行，但用户、租户成员、角色、组织成员和属性的最终写入必须全成或全败。
- **FR-020**：页面必须持续展示确认失败原因和后台进度，允许关闭后恢复；状态查询不得重复返回全部万级行数据。
- **FR-021**：初始密码明文不得因异步处理写入数据库、日志、响应或审计；每个新用户仍必须使用独立的安全密码摘要。
- **FR-022**：后台 Worker 必须把空队列视为正常状态，不得持续输出 `record not found` 错误；空闲探测频率必须可配置并采用低频默认值。
- **FR-023**：确认接口必须仅查询批次归属、有效期、状态和统计等轻量元数据，不得同步加载万行 `rows_json`；Worker 必须在业务事务开始前加载完整快照，并以反序列化后重新序列化的规范 JSON 校验摘要，避免 MySQL JSON 存储规范化造成误判，同时继续拒绝真实篡改。
- **FR-024**：Worker 的待领取排序与加锁查询必须使用不包含 `rows_json` 的窄行投影；领取租约提交后再按主键加载完整快照，避免大字段进入 MySQL filesort 并耗尽排序内存。
- **FR-025**：用户预校验必须拒绝“邮箱已归属于其他用户名”的行，并识别文件内重复邮箱；批量 UPSERT 后必须确认每个导入用户名都解析到非零用户主键，任何缺失都必须使整个业务事务回滚，禁止写入 `user_id=0` 关系。
- **FR-026**：租户成员管理接口必须使用服务端分页，单页关联用户和角色查询的 `IN` 参数数量必须受页大小约束，不得因租户拥有万级成员而一次加载全部成员或生成万级 ID SQL。
- **FR-027**：系统必须提供幂等迁移清理历史 `user_id=0` 导入关系；迁移只删除无法对应任何真实用户的零主键关系，不得按用户名、邮箱或模糊条件删除业务数据。

### 增量成功标准

- **SC-010**：合法批次点击确认后，管理员在 1 秒内看到“已受理”或明确错误，不再出现无反馈点击。
- **SC-011**：10000 行批次处理期间，页面进度信息更新间隔不超过 2 秒，关闭并重新进入后能够恢复。
- **SC-012**：同一批次连续确认 3 次仍只产生一次业务导入。
- **SC-013**：万级用户导入不再产生与行数线性倍增的逐行查询链路，数据库交互以有界批次执行。
- **SC-014**：任务失败、进程重启或重复领取均不会留下部分业务写入或泄露密码材料。
- **SC-015**：空队列持续运行 1 分钟时不产生 `record not found` 日志，默认数据库探测次数不超过 12 次。
- **SC-016**：10000 行合法批次的确认请求不传输或解析逐行快照，并在 1 秒内返回已受理；MySQL 改变 JSON 键顺序或空白时 Worker 仍可通过快照校验，字段值发生变化时必须失败。
- **SC-017**：包含万行 JSON 快照的排队批次能够被 Worker 成功领取，领取排序 SQL 只选择主键且不出现 `Out of sort memory`，状态进入 `IMPORTING` 后尝试次数至少为 1。
- **SC-018**：邮箱与其他用户名冲突的导入行在预览阶段得到可定位错误；即使验证后出现并发唯一键冲突，正式导入也会整体回滚且所有关系表均不存在 `user_id=0`。
- **SC-019**：拥有 10000 名成员的租户打开成员页时，后端默认只返回 50 条和总数，角色查询的用户 ID 不超过当前页大小，前端可翻页且不渲染万行 DOM。
